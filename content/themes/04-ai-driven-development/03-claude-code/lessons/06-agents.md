サブエージェントを活用した並列・協調作業のしくみを学びます。通常の Claude Code との違い、標準で用意されているエージェントの種類、カスタムエージェントの作り方、実践のコツまでを順に見ていきます。

## サブエージェントとは何か

### 通常の Claude Code との違い

通常の Claude Code は、ユーザーとの 1 対 1 の会話の中でコードを読み書きしたり、コマンドを実行したりします。これは **シングルスレッド** (一人で順番に作業をこなす) のイメージです。

一方、サブエージェント機能を使うと、Claude Code は **メインエージェント** として全体を指揮しながら、個々の作業を **サブエージェント** と呼ばれる専門の担当者に振り分けられます。一人の担当者が大きな仕事を複数の部下に割り振るようなイメージです。

### サブエージェントを使う 3 つのメリット

| 観点 | 内容 |
| :--- | :--- |
| 並列処理 | 複数のサブエージェントが同時に別々の作業を進められ、調査時間を大幅に短縮できます。 |
| 専門性の活用 | タスクの種類に応じて、コード探索・実装計画などに最適なエージェントを選べます。 |
| コンテキスト保護 | 大量の調査結果をサブエージェントに任せることで、メインのコンテキストウィンドウをきれいに保てます。 |

### メインエージェントとサブエージェントの関係

メインエージェント (Claude) が全体を指揮し、必要に応じて目的ごとのサブエージェントを起動します。標準で利用できる代表的なエージェントは次のとおりです。

- 汎用 (`general-purpose`): 幅広いタスク全般
- 探索 (`Explore`): コードベースの検索と読解
- 計画 (`Plan`): 読み取り専用の設計・実装計画
- 専門エージェント (その他): セキュリティ評価・パフォーマンス改善などの特定領域

各サブエージェントは **独立したコンテキスト** で動作し、結果をメインエージェントに報告します。これによりメイン側の会話履歴を汚さずに、大量の調査や検証をこなせます。

### サブエージェントが起動される場面

- **複雑な調査タスク** -- 「このプロジェクト全体の API エンドポイントを調べて」のような、複数ファイルにまたがる広範な調査が必要な場合
- **段階的な実装** -- 「認証機能を設計し、実装し、テストまでを一気に行って」のように、複数のフェーズに分かれる大きな実装タスク
- **並列で実行できる作業** -- 「A 機能と B 機能をそれぞれ調べて比較して」のように、互いに独立した作業を同時進行できる場合

## サブエージェントの種類

Claude Code には目的に応じた複数種類のサブエージェントが存在します。メインの Claude Code が状況を判断し、もっとも適切なエージェントを選んで起動します。

| 種類 | 英語名 | 用途 | 特徴 |
| :--- | :--- | :--- | :--- |
| 汎用エージェント | `general-purpose` | 幅広いタスク全般 | コード調査・ファイル検索・複数ステップの作業に対応。専門エージェントに当てはまらない一般タスクを担当 |
| 探索エージェント | `Explore` | コードベース探索 | パターン検索やキーワード検索に特化。`thoroughness` パラメータで探索の深さ (quick / medium / very thorough) を制御できる |
| 計画エージェント | `Plan` | 設計・調査・実装計画 | 読み取り専用。どのファイルをどの順序で変更するかといったアーキテクチャ上の判断を担当 |
| その他の専門エージェント | -- | 特定領域の専門作業 | セキュリティ評価・パフォーマンス改善など多数の専門エージェントが存在する |

### 汎用エージェント (general-purpose)

コードの調査、ファイルの検索、複数ステップの作業など幅広いタスクに対応します。特定の専門エージェントに当てはまらない作業を担当します。

- **自動起動のタイミング**: 特定の専門エージェントに当てはまらない一般的なタスクが発生したとき
- **使用例**: 「このコードの処理を説明して」のように、探索・計画・実行の区別がない汎用的な作業

### 探索エージェント (Explore)

ファイルのパターン検索やコード内のキーワード検索に特化します。`thoroughness` パラメータで探索の深さを制御でき、ざっと調べたいときから網羅的に調査したいときまで使い分けられます。

- **自動起動のタイミング**: 広範なコードベースの理解が必要なとき、「この API はどう動いているか」といった読解タスク
- **使用例**: 「このプロジェクトの認証フローを調べて」-- Explore が関連ファイルを網羅的にスキャン

### 計画エージェント (Plan)

読み取り専用のエージェントで、コードの変更は行いません。「どのファイルを変更すればよいか」「どんな順序で実装を進めるべきか」といったアーキテクチャ上の判断を担当します。

- **自動起動のタイミング**: 複雑な変更の事前調査や影響範囲の分析が求められるとき
- **使用例**: 「TypeScript への移行計画を立てて」-- Plan が既存コードを分析して段階的な移行プランを提示

> [!TIP]
> `Explore` の `thoroughness` は、`quick` なら高速な概要把握に、`very thorough` なら時間をかけた網羅調査に向いています。状況に応じて指定しましょう。

## サブエージェントの動き方

### 基本的な流れ (5 ステップ)

たとえば「このプロジェクトのバグを見つけて修正して」と指示すると、Claude Code はおおよそ次の流れで動きます。

1. **タスクの分析** -- メインエージェントがタスクの全体像を把握し、必要な作業を整理します。
2. **サブエージェントの起動** -- 「コードを広く調査する」作業を探索エージェントに振り分けます。
3. **並列実行** -- 複数の調査が必要な場合、複数のサブエージェントを同時に起動して並列で作業させます。
4. **結果の統合** -- 各サブエージェントの調査結果をメインエージェントが受け取り、統合して最終判断を下します。
5. **実装と確認** -- 調査結果をもとに修正を加え、問題が解決されたかを確認します。

### バックグラウンド実行

サブエージェントはバックグラウンドで実行することもできます。メインエージェントが別の作業をしている間に、サブエージェントが調査を進めるイメージです。バックグラウンドのサブエージェントは完了次第、結果をメインエージェントに報告します。

### ワークツリーによる分離

大きな変更を伴うタスクでは、「ワークツリー」と呼ばれる仕組みを使って、サブエージェントが元のコードとは分離された作業コピー上で作業することがあります。これにより、サブエージェントの変更が即座に本体のコードに影響することを防ぎ、安全に実験的な作業ができます。**変更がない場合、ワークツリーは自動的に削除されます。**

## ユーザーとして意識すべきこと

サブエージェント機能の多くは Claude Code が自動で判断して動かしますが、ユーザーとして知っておくべきポイントがあります。

### 承認が必要な操作

サブエージェントがファイルの編集やコマンドの実行を行う場合、通常の Claude Code と同様にユーザーへの確認が入ります。サブエージェントだからといって確認なしに勝手に動き回ることはありません。安全性の仕組みは維持されています。

### 大きなタスクは明示的に伝える

「このプロジェクト全体を調べて」「複数のファイルを一括修正して」のような大きなタスクを指示すると、Claude Code はサブエージェントを活用する可能性が高くなります。細かいタスクを一つずつ指示するよりも、大きな目標をまとめて伝えたほうが効率的な作業計画を立てやすくなります。

### 処理時間が長くなることがある

サブエージェントが並列で作業している場合、ユーザーから見ると Claude Code が黙って処理し続けているように見えることがあります。これは正常な動作で、バックグラウンドで作業が進んでいます。

### コンテキストとコストへの影響

サブエージェントも内部的に API を呼び出します。そのため、サブエージェントを多用するタスクはトークンの消費量が増えます。`/cost` コマンドで消費量を確認しながら作業するのが望ましいです。

> [!IMPORTANT]
> サブエージェントを多用する作業は、処理時間とトークン消費量が大きくなりがちです。大きなタスクを任せるときは、時間に余裕のあるタイミングで行い、`/cost` で定期的にコストを確認しましょう。

## 実践的な使い方のコツ

### 1. まず `/plan` で計画を確認してから実行する

サブエージェントを使う際の基本的なワークフローです。まず `/plan` で目標を伝えて計画を立てさせ、内容を確認してから実行に移ります。「意図と違う方向に進もうとしていないか」「変更対象のファイルに見落としはないか」をここで確認できます。

```text
> /plan 既存の REST API を GraphQL に移行したい
```

### 2. 要件が明確な調査タスクはまとめて渡す

調査のみを目的とする (コードを変更しない) タスクは、まとめて渡してサブエージェントの並列処理を活かすのが効果的です。調査はファイルを変更しないため認識のずれによるリスクが低く、複数のサブエージェントが並列で調べることで時間を短縮できます。

```text
> このプロジェクトの認証まわりのコードをすべて調査して、
  セキュリティ上の問題点があれば一覧にして報告して
```

### 3. 要件が曖昧なときは 1 ステップずつ進める

「何となく改善したい」「どこが問題かまだわからない」という段階では、大きなタスクをいきなり渡すのは避けましょう。まずは小さな調査から始めて状況を把握し、方針が固まってから `/plan` で計画を立てる流れが安全です。

```text
> src/auth/ の中にあるファイルの構成と役割を説明して
```

### 4. 進捗を確認しながら作業する

サブエージェントを使った大きなタスクでは、途中経過を確認するのが効果的です。完了した作業と残りの作業を把握でき、途中で方針を変えたい場合にも早めに対処できます。

```text
> 現在どこまで作業が進んでいるか教えて
```

## カスタムサブエージェントの作成

ビルトインのサブエージェント以外にも、プロジェクト固有の専門エージェントをマークダウンファイルで定義できます。カスタムエージェントを登録しておくと、Claude Code はタスクの内容と各エージェントの `description` を照合し、もっとも適切なエージェントを自動で選んで起動します。

### `/agents` コマンドで簡単作成

マークダウンファイルを手書きしなくても、`/agents` コマンドを使えば対話形式でカスタムエージェントを作成できます。Claude Code がどんなエージェントを作りたいかを質問し、ファイルを自動生成してくれます。

```text
> /agents
# → 対話形式でエージェントの名前・目的・使用ツールを入力
# → .claude/agents/<name>.md が自動で生成される
```

### ファイルの置き場所 (手動作成の場合)

| パス | スコープ | 説明 |
| :--- | :--- | :--- |
| `.claude/agents/` | プロジェクト固有 | そのプロジェクト内でのみ使えるカスタムエージェント。Git で管理すればチームで共有できます |
| `~/.claude/agents/` | グローバル | すべてのプロジェクトで使えるエージェント |

### ファイルの書き方

カスタムエージェントのファイルは、フロントマター (ファイル先頭のメタ情報) と本文で構成されます。`description` の内容は、Claude Code がタスクの内容に応じて「このエージェントを使うべきか」を判断する際に参照されます。

```yaml
---
name: code-reviewer
description: コードレビューを行う専門エージェント。
  PR の diff を受け取り、品質・セキュリティ・パフォーマンスの観点でレビューコメントを出力する。
---
あなたはコードレビューの専門家です。以下の観点でレビューを行ってください。

## レビュー観点

**品質**
- 変数名や関数名は意図が伝わるか
- 同じ処理が重複していないか
- 関数が単一の責任を持っているか

**セキュリティ**
- SQL インジェクションや XSS の可能性はないか
- 認証・認可の処理は適切か
- 機密情報がログに出力されていないか

**パフォーマンス**
- 不必要なループや再計算がないか
- データベースへの問い合わせが最適化されているか

各問題点は重要度 (高・中・低) とともに報告してください。
```

### 使えるツールを制限する

フロントマターの `tools` フィールドを使うと、そのエージェントが使えるツールを限定できます。ドキュメント担当のエージェントがうっかりコードを書き換えてしまうような事故を防げます。

```yaml
---
name: doc-writer
description: ドキュメントの作成・更新に特化したエージェント。
  コードの読み取りとドキュメントファイルの編集のみを行う。
tools: Read, Write, Edit
---
あなたはドキュメント作成の専門家です。
コードを読み取り、わかりやすいドキュメントを作成します。
コマンドの実行やコードの変更は行いません。
```

### カスタムエージェントの活用例

プロジェクトの特性に合わせたカスタムエージェントを用意しておくと、繰り返し発生する定型作業を自動化できます。

- **テスト生成エージェント**: 関数やクラスのコードを読み込み、そのプロジェクトのテスト記述スタイルに合わせたテストコードを生成する
- **コミットメッセージ生成エージェント**: 変更差分を分析し、プロジェクトのコミットメッセージ規約に従ったメッセージを生成する
- **依存関係チェックエージェント**: `package.json` や各種設定ファイルを調べ、古くなったパッケージや脆弱性を持つ依存関係を報告する

> [!TIP]
> 社内ルールに沿ったレビューや、リリース前のチェックリスト確認など、**再現性の高い定型作業** ほどカスタムエージェントの効果が出ます。まず 1 つ作って運用し、必要に応じて観点や許可ツールを調整しましょう。

## サブエージェントの限界と注意点

サブエージェントは強力な機能ですが、万能ではありません。

> [!WARNING]
> **複雑すぎるタスクへの対応**: あまりにも曖昧な指示や、人間的な判断が必要な場面では、サブエージェントが期待どおりの結果を返せないことがあります。大きなタスクを渡す場合でも、要件や制約を具体的に伝えることが重要です。

> [!WARNING]
> **承認の連続**: サブエージェントが多くのファイル操作を行う場合、ユーザーへの確認が複数回続くことがあります。大きなタスクを指示する際は、時間に余裕があるときに行うのが望ましいです。

> [!CAUTION]
> **意図しない変更のリスク**: サブエージェントが広範にファイルを調査・編集する場合、意図しないファイルが変更されるリスクがゼロではありません。重要な作業の前には必ず git でコミットしておき、いつでも元に戻せる状態を作っておきましょう。

## Agent Teams と Cowork

複数のエージェントをチームとして編成する **Agent Teams** や、エージェントと人間が同じタスクを共同で進める **Cowork** のような発展的な使い方も提案されています。基本となるサブエージェントと `/plan` の使い方に慣れてから、必要に応じて学ぶのがおすすめです。

## まとめ

- **サブエージェント** は、複雑なタスクを処理するために呼び出される専門の AI エージェントです。並列処理・専門性の活用・コンテキストの保護というメリットがあります。
- 汎用・探索・計画エージェントなどタスクに応じた専門家が存在し、Claude Code が状況を判断して自動選択します。
- 活用のコツは **大きなタスクをまとめて渡す** / **`/plan` で計画を立ててから進める** / **進捗を適宜確認する** / **重要な変更の前は git でコミットしておく** の 4 点です。
