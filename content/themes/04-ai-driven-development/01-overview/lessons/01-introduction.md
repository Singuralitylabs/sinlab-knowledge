# AIコーディング/AI駆動開発とは何か

2024 年以降、ソフトウェア開発の現場は「人間がコードを書き、AI が補助する」段階から、**「人間が意図を伝え、AI がコードに翻訳する」段階** へと移行し、2026 年現在では **AI エージェントの動作環境そのものを設計する段階** に入りつつあります。本章では、このパラダイムシフトである **AI 駆動開発（AI-Driven Development）** の定義と、従来の開発との違い、そして使いこなすための勘所を整理します。

## このページで学べること

- **AI コーディング / AI 駆動開発** の定義と、単なるコード補完との違い
- 現行 AI コーディングツールの **3 つの形態**（チャット型 / IDE 統合型 / ターミナル型）
- コード補完から **ハーネスエンジニアリング** に至る進化の系譜
- 従来開発との違い、**求められるスキルの重心の変化**
- 変わること / 変わらないこと
- 活用上の **3 つの注意点**（ハルシネーション / リソースの有限性 / 理解なき使用）

## AI 駆動開発とは

AI 駆動開発とは、**AI と協働してソフトウェアを開発するアプローチ** の総称です。従来の「人間がコンピュータに命令を書く」行為から、「人間が AI に意図を伝え、AI がコードに翻訳する」行為への転換を指します。

重要なのは、これが単なる「速いコード補完」ではない点です。設計の相談、実装、デバッグ、ドキュメント作成、リファクタリング、コードレビュー—— 開発のあらゆる場面で AI が **パートナー** として機能することが特徴です。

### コード補完との違い

従来のコード補完は「タイピングの省略」が本質でした。AI 駆動開発では、AI は単語やトークンではなく **開発プロセスそのもの** に参加します。

| 観点 | 従来のコード補完 | AI 駆動開発 |
| --- | --- | --- |
| 支援の粒度 | 行・トークン単位 | ファイル・機能単位 |
| 関与する工程 | 実装時のみ | 設計〜実装〜テスト〜ドキュメント |
| 人間の役割 | 書く | 意図を伝え、評価・修正する |
| 必要な入力 | コードの一部 | 自然言語での要件・文脈 |

## AI コーディングツールの 3 つの形態

現行の AI コーディングツールは、使われ方の違いから以下の 3 つに大別できます。

![AIツールの3つの形態：チャット型 / IDE統合型 / ターミナル型](/content-assets/04-ai-driven-development/01-overview/images/introduction/tool-types.png)

### チャット型

Claude / ChatGPT / Gemini などの汎用チャットインターフェース。ブラウザで質問して答えを得る形式です。

- **強み** — 導入が簡単、どんな言語・テーマでも相談可能
- **弱み** — コードの適用は手作業でコピペが必要、プロジェクト全体の文脈を持ちにくい

### IDE 統合型

Cursor / GitHub Copilot / Windsurf など、エディタに組み込まれたタイプ。コードを書きながらインライン補完や対話ができます。

- **強み** — 既存のワークフローに溶け込む、開いているファイルを自然に参照できる
- **弱み** — エディタに縛られる、プロジェクト横断の大規模作業は不得手

### ターミナル型

Claude Code / Codex CLI / Gemini CLI などのコマンドライン型。ファイルシステム・シェル・Git に直接アクセスし、**エージェント的に作業を進められる** のが特徴です。

- **強み** — ファイル横断の編集、コマンド実行、長時間の自律作業が可能
- **弱み** — 学習コストが高く、CLI 操作に慣れが必要

> [!TIP]
> 3 つは排他ではなく、**組み合わせて使う** のが現実的です。気軽な相談はチャット型、通常の実装は IDE 統合型、大規模なリファクタや機能追加はターミナル型 —— といった使い分けが定着しつつあります。

## 進化の系譜

AI によるコーディング支援は、おおむね以下の段階を経て進化してきました。

![AI支援の進化プロセス：フェーズ1（従来の補完）/ フェーズ2（AI補完）/ フェーズ3（AI駆動開発）](/content-assets/04-ai-driven-development/01-overview/images/introduction/evolution-phases.png)

1. **従来型補完（〜 2021 年）** — 静的解析ベースの補完。辞書引きとタイピング省略が主な役割
2. **AI 補完（2021 〜 2023 年）** — GitHub Copilot が代表例。周辺コードの文脈を読み取り、次に書きそうな行を提案するレベル
3. **AI 駆動開発（2024 〜 2025 年）** — Claude Code / Cursor などが象徴。ファイル横断で編集し、テストを実行し、ドキュメントを書くなど **プロセス全体に参加** するレベル

### フェーズ 4: ハーネスエンジニアリング（2026 年 〜）

2026 年現在、主戦場は **AI モデルそのもの** から **AI の動作環境（ハーネス）を設計すること** へと移りつつあります。同じモデルでも、周辺のツール・メモリ・権限・ワークフローを適切に設計すれば生産性は桁違いに変わる、という認識が定着してきました。

ハーネスエンジニアリングの主要な構成要素は以下の通りです。

- **コンテキスト設計** — CLAUDE.md / Rules / メモリ機能で AI に渡す前提情報を最適化する（→ [コンテキストエンジニアリング](/themes/04-ai-driven-development/01-overview/context-engineering)）
- **ツール拡張（MCP）** — Model Context Protocol により、AI に外部サービス・DB・ブラウザなどを操作する手段を与える
- **サブエージェント** — タスクごとに独立したコンテキストでエージェントを走らせ、並列化・役割分担する
- **Skills / スラッシュコマンド** — 定型的な作業手順をコマンド化し、AI のワークフローとして再利用可能にする
- **Hooks** — AI の動作の前後に自動処理（フォーマット・テスト・通知など）を挟み、ガードレールを敷く
- **権限制御** — 自律実行の範囲を明示的に設計し、安全と速度を両立させる

> [!NOTE]
> フェーズ 3 が「AI に仕事をさせる」技術の成熟だったのに対し、フェーズ 4 は **「AI に仕事をさせる環境を設計する」** 技術の時代です。プログラマーの仕事は「コードを書く」から「コードを書く AI のための仕組みを作る」へと、さらに一段メタな方向へシフトしています。

## 従来開発との違い

AI 駆動開発では、開発の起点・調査方法・デバッグの進め方まで、あらゆる局面が変化します。

| 観点 | 従来の開発 | AI 駆動開発 |
| --- | --- | --- |
| 起点 | コードを書くこと | 意図を伝えること |
| 調査 | ドキュメントを探して読む | AI に質問して整理させる |
| デバッグ | 一人で追跡 | AI と協働で仮説検証 |
| 学習 | 段階的に習得 | 作りながら実践的に習得 |
| 作業単位 | 行・関数 | ファイル・機能 |

### 求められるスキルの重心が動く

単に「楽になる」のではなく、**求められるスキルの比重が変化** する点に注意が必要です。

- 重要度が **上がる** スキル — 意図の言語化、仕様の分解、生成物の評価・修正、アーキテクチャ設計、ドメイン知識
- 重要度が **相対的に下がる** スキル — 構文暗記、定型実装、単純な辞書引き的な知識

> [!NOTE]
> 「プログラミング不要になる」ではなく、「プログラミングの **どこに時間を使うか** が変わる」と捉えるのが実態に近いです。構文を打鍵する時間が減る代わりに、要件整理・設計・レビューの時間が増えます。

## 変わること / 変わらないこと

AI 駆動開発によって実装速度や学習方法は大きく変わりますが、**ソフトウェア開発の本質は変わりません**。

| カテゴリ | 変わること | 変わらないこと |
| --- | --- | --- |
| 実装 | 定型コードの生産速度が大幅に向上 | 複雑な設計判断は人間が担う |
| 学習 | 作りながら学ぶスタイルが主流化 | 基礎概念を理解する重要性 |
| 責任 | コードの **書き手** が変わる | コードの **品質責任** は人間側 |
| 知識 | 構文・API の暗記は不要に | ドメイン知識・業務理解は必須 |

## 活用上の 3 つの注意点

AI 駆動開発の恩恵を受けるには、AI の「苦手」を理解しておく必要があります。特に以下の 3 点は、初心者が最もハマりやすい落とし穴です。

### 1. ハルシネーション（もっともらしい嘘）

AI は「次に来る確率の高い単語」を予測する仕組みであり、**存在しない関数・ライブラリ・API を自信満々に提案してくる** ことがあります。

- **対策** — 生成されたコードを必ず動作確認する、公式ドキュメントで API の実在を確認する、違和感があれば AI に根拠を尋ねる

### 2. リソースの有限性

AI には **コンテキストウィンドウ**（一度に扱える情報量）、**トークン制限**、**レート制限**、**コスト** などの有限性があります。制限を超えると、以前の指示を忘れたり、応答精度が落ちたりします。

- **対策** — プロンプトを簡潔に保つ、長いタスクは分割する、新しい話題では会話をリセットする
- 詳細は [コンテキストエンジニアリング](/themes/04-ai-driven-development/01-overview/context-engineering) を参照

### 3. 理解なき使用の危険性

最も本質的な落とし穴は **「動くコード ≠ 良いコード ≠ 理解したコード」** という事実です。AI の生成物を読まずにマージしていると、以下の問題に直結します。

- バグが出たときに対応できない
- 要件変更に追従できず、拡張が困難になる
- セキュリティ脆弱性を見落とす
- AI なしでは一行も書けない状態に陥る

一方で、AI 時代に **「すべてのコードを行単位で理解してからマージ」** を厳格に徹底するのは非現実的です。生成量が多いほど読み切れず、「AI を使う意味」と衝突します。ライブラリのソースを全部読まずに使うのと同じで、**コードの重要度・寿命に応じて、求める理解の粒度を使い分ける** のが現実的です。

| 区分 | 必要な理解の粒度 |
| --- | --- |
| 本番・長期運用・クリティカルパス | **行単位** で理解、疑問があれば AI に説明させる |
| 一般的な機能実装 | **インターフェース・振る舞い** を理解、テストで担保 |
| プロトタイプ・スパイク・使い捨て | **動作確認** で OK、必要になった時点で深掘り |

> [!CAUTION]
> 粒度は使い分けつつも、以下の最低ラインは必ず守りましょう。
> - **取り返しのつかない箇所**（セキュリティ・認証・決済・データ破壊系）は区分に関わらず **行単位で理解する**
> - **「自分がどこを理解していないか」を把握する**（= 理解していない領域の存在を自覚する）
> - **テストを書き、動作を検証可能な状態** にしておく。理解できていなくても、壊れたら気付ける状態を作ることが最低条件

## AI との理想的な関係

AI は魔法の箱ではなく、**「優秀だが前提知識を持たないジュニアエンジニア」** と捉えるのが適切です。

- **丸投げしない** — 「良い感じに作って」では良いコードは出ない
- **指示・レビュー・フィードバックのループを回す** — 人間のチームメンバーと同じ付き合い方を適用する
- **責任は自分が持つ** — 最終的な品質責任を AI には委譲できない

この関係性を意識できれば、AI 駆動開発は単なる効率化ツールではなく、**個人の開発能力を何倍にも拡張する協働者** になります。

## まとめ

- AI 駆動開発は、コード補完の延長ではなく **開発プロセスそのものへの AI の参加** である
- ツールは **チャット型 / IDE 統合型 / ターミナル型** の 3 形態を使い分けるのが基本
- 2026 年現在は **ハーネスエンジニアリング** の段階に移行中。AI の動作環境を設計する技術が生産性を分ける
- 求められるスキルは **構文知識から「意図の言語化・評価・設計」へ** 重心が移る
- 本質（設計思考・品質責任・ドメイン知識）は変わらない
- 理解の粒度はコードの重要度で使い分けつつ、**取り返しのつかない箇所は必ず行単位で理解** する
- ハルシネーション・リソース制約・理解なき使用の 3 つに注意しつつ、**AI を優秀なジュニアエンジニアとして扱う** ことが成功の鍵
