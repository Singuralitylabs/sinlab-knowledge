# 開発ワークフローはどう変わるか

AI 駆動開発のインパクトは「コードを速く書ける」だけにとどまりません。**要件定義からデプロイ後の運用まで、ソフトウェア開発のあらゆるフェーズで仕事の重心が動きます**。本章では、従来の開発プロセスと AI 駆動開発を対比しながら、各フェーズでどんな変化が起きているのか、そしてその変化にどう向き合えばよいのかを整理します。

## このページで学べること

- AI 駆動開発における **新しい開発ループ** の全体像
- 設計・実装・レビュー・テスト・運用の **各フェーズで起こる具体的な変化**
- **仕様書駆動開発（スペック駆動）** と **Plan → Execute** パターンの位置付け
- ペアプログラミングからの進化形である **AI ペア開発** の勘所
- ワークフロー再設計時に **陥りがちな落とし穴** と回避方法

## 全体像：単線のウォーターフォールから「螺旋ループ」へ

従来の開発は、要件 → 設計 → 実装 → テスト → リリース と **単線で進む** イメージで語られてきました。アジャイルやスクラムでもスプリントごとに同じ流れを回すだけで、各フェーズの内部構造は単線のままです。

AI 駆動開発では、各フェーズが **「人間 ↔ AI」の往復ループ** に置き換わります。1 つのタスク内でも、**意図伝達 → 生成 → 評価 → 修正指示 → 再生成** が何度も繰り返されます。

![従来開発と AI 駆動開発のフロー比較](/images/ai-driven-development/workflow-changes/flow-comparison.png)

| 観点 | 従来の開発 | AI 駆動開発 |
| --- | --- | --- |
| 進行モデル | 単線・段階的 | 螺旋・反復的 |
| 1 タスクの所要時間 | 長め（書く時間が支配的） | 短め（書く時間 < 評価・指示の時間） |
| ボトルネック | 実装速度 | 意図の言語化と生成物のレビュー |
| 主要な活動 | コードを書く | 文脈を渡し、生成物を評価する |
| ドキュメント | 後追いで書かれがち | **AI への入力として先に整える** |

> [!NOTE]
> 「単線が螺旋になる」という変化は、生産性が単純に上がる話ではなく **回転速度を上げないと逆に遅くなる** ことを意味します。意図伝達と評価の質を高めるスキルが、新たなボトルネックになります。

## フェーズ別に見る変化

### 1. 要件定義・設計フェーズ

**従来** — 要件を聞き取り、ホワイトボードや設計書で議論し、ある程度固まってから実装に着手する。設計書は実装後に陳腐化することも多い。

**AI 駆動開発** — AI を **壁打ち相手** として早期から巻き込み、要件と設計を **対話で詰める** スタイルに変わります。

- **AI を質問者として使う** — 「このアプリの目的は」「ユーザーは誰か」「非機能要件は」と AI に聞いてもらい、抜け漏れを潰す（[コンテキストエンジニアリング](/themes/04-ai-driven-development/01-overview/context-engineering)で扱う Plan モードの活用）
- **設計のたたき台を AI に書かせる** — `requirements.md` や `architecture.md` の初稿は AI が書き、人間はレビューと加筆に専念する
- **設計とコードの距離が縮む** — 設計書がそのまま AI への指示になるため、**設計の更新が実装に直結** する

> [!TIP]
> 「設計書を書く時間が増えて遅くなる」と感じる場合、それは順序を取り違えている可能性があります。**AI に書かせて人間がレビューする** のが原則。手書きの時間を増やすのではなく、AI への入力品質を高める時間に投資します。

### 2. 実装フェーズ

**従来** — 設計を見ながら IDE でコードを書く。新しいライブラリは公式ドキュメントを開いて読みながら使う。

**AI 駆動開発** — **「意図を渡してコードを受け取る」** が基本動作になります。打鍵時間は減り、その分が **指示・評価・修正のループ** に置き換わります。

- **Plan → Execute の二段構え** — 大きめの実装は、いきなりコードを書かせず、まず計画（変更ファイルリスト・アプローチ）を出させて合意してから実行する
- **生成 → 動作確認 → 修正指示 → 再生成** のループ — 一発で完璧なコードは出ないため、**短いサイクルで回す** 前提で進める
- **タスクの粒度が変わる** — 1 行・1 関数単位ではなく、**機能単位・PR 単位** で AI に依頼するのが現実的
- **ボイラープレートは AI 任せ** — フォームバリデーション、CRUD エンドポイント、テストの雛形などは指示一発で出る

> [!NOTE]
> 第 4 世代以降のターミナル型エージェント（Claude Code / Codex CLI / Gemini CLI）では、**ファイルの読み書き・コマンド実行・テスト実行までを AI が連続で行う** ため、「AI に任せて、人間は別のタスクに移る」並列作業が可能になっています。

### 3. レビューフェーズ

**従来** — 人間が書いたコードを別の人間がレビューする。レビュアーは「書き手の意図」を推測しながら読む。

**AI 駆動開発** — レビュー対象の主役が **「AI が書いたコード」** に変わります。書き手（AI）は説明に応じてくれる一方、**人間レビュアーの責任は重くなります**。

- **コードの品質責任は人間** — 「AI が書いたから自分は知らない」は通用しない。マージしたら自分のコード
- **AI レビュアーとの併用** — Copilot Code Review など、PR を **AI に一次レビューさせる** 仕組みが定着している。人間は AI の指摘を仕分け、本質的な観点に集中する
- **diff を読む筋力が問われる** — 一度に出力される変更量が増えるため、**差分を素早く把握する力** が以前より重要になる
- **「なぜこうしたか」を AI に説明させる** — 不可解な実装を見つけたら、AI 自身に根拠を尋ねる。納得できなければ書き直させる

> [!CAUTION]
> AI レビューは「人間レビューの代替」ではなく **「足し算」** です。AI レビューだけで通すと、AI が苦手とする観点（業務要件の妥当性、長期的な保守性、セキュリティ境界の判断）が抜け落ちます。最終承認は必ず人間が担います。

### 4. テストフェーズ

**従来** — 実装後にテストコードを書く。テスト容易性のために設計を後追いで直すこともある。

**AI 駆動開発** — テストは **AI 駆動開発の品質ガード** として、より重要度が増します。AI が書いたコードを **「動くこと」だけを根拠に信用するのは危険** だからです。

- **テストを先に AI に書かせる** — 仕様を渡してテストを生成 → そのテストが通るように実装、という TDD 的な進め方が AI と相性が良い
- **エッジケースの列挙を AI に依頼する** — 人間が思いつかない境界条件を網羅的に出してもらう
- **失敗時の自己修正ループ** — テストが失敗すると、AI が原因を分析して修正する流れを回せる（Claude Code などでは `bun test` などを AI 自身が実行・確認できる）
- **回帰テストの維持コストが下がる** — テスト追加・修正の打鍵コストが下がるため、**テストを書かない言い訳が減る**

> [!TIP]
> 「AI が書いたコードを AI が書いたテストで担保する」だけでは独り相撲になりかねません。**テストの観点（何を検証すべきか）は人間が決める**、テストの実装は AI に任せる、という役割分担が安全です。

### 5. ドキュメンテーション

**従来** — ドキュメントは「書かれない」「書かれても陳腐化する」が常態だった。

**AI 駆動開発** — ドキュメントは **AI への入力（コンテキスト）として価値を持つ** ため、書かれる動機が強くなります。

- **コードからドキュメントを生成** — 関数コメント、API 仕様、README、CHANGELOG などを AI が初稿を作成
- **ドキュメントが実装を駆動する** — `docs/` 配下のファイルが AI への入力になることで、**書くと AI が賢くなる** という即時の見返りがある
- **更新も AI が担当** — タスク完了時に `tasks.md` を更新するなど、AI 自身に同期させる運用が広がっている

詳細は [コンテキストエンジニアリング](/themes/04-ai-driven-development/01-overview/context-engineering) の仕様書駆動開発を参照してください。

### 6. 運用・保守フェーズ

**従来** — 障害対応・ログ調査・原因特定は **属人化** しやすく、ベテラン依存になりがち。

**AI 駆動開発** — ログ・スタックトレース・コード・ドキュメントを **AI に渡して原因を仮説立てさせる** 流れが定着しつつあります。

- **障害解析の壁打ち** — ログとエラーメッセージを貼り付けて、AI に仮説を出してもらう
- **リファクタリングの心理的ハードルが下がる** — 大規模な書き換えも AI に任せられるため、**負債を放置しなくなる**
- **オンコール文書の生成** — 障害ごとのポストモーテムやランブックを AI が初稿作成
- **ただし本番操作の権限は別問題** — 「AI が原因を特定」と「AI が本番に手を入れる」は別物。後者は **権限制御とレビュー** が必須

> [!CAUTION]
> 本番環境への変更を AI が自律実行することには **明示的な権限設計** が必要です。Claude Code の `permissions` 設定、Hooks による事前検証、`--dangerously-skip-permissions` の取り扱いなど、ガードレールを意識した運用設計が前提となります。

## 新しいワークフローパターン

### Plan → Execute パターン

「いきなり書かない」が AI 駆動開発の鉄則です。

1. ゴールと制約を AI に伝える
2. **計画モード**（Claude Code の Plan モード）や、変更を diff で先にプレビューする機能（Cursor の Composer など）で **変更ファイル・アプローチ** を確認する
3. 計画にレビューを入れて合意する
4. 合意した計画を実行させる

> [!TIP]
> 「計画段階で十分な議論をする」ほうが、コード生成後に修正させるより圧倒的に手戻りが少なくなります。**AI のトークンと自分の時間を節約する最大のコツ** です。

### 仕様書駆動（スペック駆動）開発

要件・設計・タスク分解をすべて Markdown のドキュメントに落とし、AI と人間がそれを見ながら実装を進める手法です。詳細は [コンテキストエンジニアリング](/themes/04-ai-driven-development/01-overview/context-engineering) を参照してください。

- 中・大規模プロジェクトでは **必須** に近い
- 小規模でも、**フェーズをまたぐ作業** や **複数日にわたる実装** には有効

### AI ペア開発

人間同士のペアプログラミング（ドライバ役とナビゲータ役）を **AI と人間のペア** に置き換えるスタイルです。

- **人間がナビゲータ、AI がドライバ** — 一般的な実装作業はこの構成
- **人間がドライバ、AI がナビゲータ** — 学習中・複雑な設計判断・セキュリティ要件のある箇所は逆転させる
- **常に対話の相手がいる状態** — 一人で詰まる時間が大幅に減る

### マルチエージェント協調

タスクを分割して **複数のサブエージェントに並列で進めさせる** パターンです（[コンテキストエンジニアリング](/themes/04-ai-driven-development/01-overview/context-engineering) の「分離（Isolate）」の実践）。

- リサーチ係、実装係、レビュー係、テスト係を **コンテキスト分離** して走らせる
- メインのエージェントは **サブの成果を要約として受け取る**
- 単一の長大なコンテキストでやるより **精度・速度ともに有利**

## 何が変わり、何が変わらないか

| 領域 | 変わること | 変わらないこと |
| --- | --- | --- |
| 設計 | AI と対話しながら詰める | **意思決定の責任は人間** |
| 実装 | 打鍵時間が激減、生成 → 評価のループに | アーキテクチャ判断・トレードオフ評価 |
| レビュー | AI 一次レビュー + 人間最終承認 | **マージ責任は人間** |
| テスト | テスト生成・修正の負担が大幅減 | テスト観点の設計は人間 |
| 運用 | 障害解析・ドキュメント生成を AI が支援 | 本番操作の権限・最終判断は人間 |
| ドキュメント | AI が初稿を書き、人間がレビュー | ドメイン知識・前提条件の補記 |

共通しているのは、**「判断・責任・ドメイン知識」は人間に残る** ことです。手を動かす量は減っても、**意思決定の質に対する要求はむしろ上がります**。

## 陥りがちな落とし穴

### 1. 「AI 任せの放牧」

AI に丸投げして手元を見ないと、**気づいたら自分が読めないコードベースが完成** します。マージごとに自分の理解で言語化できるか確認しましょう。

### 2. ドキュメント整備を後回しにする

「動いているからいい」とドキュメントを後回しにすると、AI のコンテキストが貧弱になり **長期的な精度が落ちます**。コードと同じ重みでドキュメントも更新する習慣が必要です。

### 3. レビューを AI だけに任せる

AI レビューは速くて広範囲をカバーできますが、**業務要件の妥当性や長期保守性** などは見落としがちです。最終承認は人間が担うラインを引きます。

### 4. プロセスを変えずにツールだけ入れ替える

Cursor や Claude Code を導入しても、**「設計せずに書く」「テストを書かない」** といった既存のクセが残っていると効果は限定的です。**ワークフローそのものを Plan → Execute 型に作り変える** 覚悟が必要です。

### 5. コンテキストを膨らませすぎる

「全部読ませれば賢くなる」は誤りです。コンテキストには有限の容量があり、**ノイズが増えると精度が下がります**。必要な情報だけを選択（Select）する規律が重要です。

> [!IMPORTANT]
> ワークフローの再設計は **個人の習慣だけでは完結しません**。チームで Plan モードの活用、PR テンプレート、AI レビューの運用ルール、`docs/` の運用などを **合意して同期する** 必要があります。AI 駆動開発の本質は個人技ではなく **チームのワークフロー再設計** です。

## まとめ

- AI 駆動開発では、**単線のウォーターフォールが螺旋ループに置き換わる**。1 タスクの中で生成 → 評価 → 修正が何度も回る
- 各フェーズで重心が変わる：設計は対話で詰め、実装は Plan → Execute で進め、テストは AI に書かせ、運用解析も AI に壁打ちする
- 新しい標準パターンは **Plan → Execute / 仕様書駆動 / AI ペア開発 / マルチエージェント協調** の 4 つ
- 何を作るかの **判断と責任は人間に残る**。打鍵量は減っても、意思決定の質は上がる
- 落とし穴は「放牧」「ドキュメント後回し」「AI レビューのみで承認」「ツールだけ入れ替え」「コンテキストの肥大化」の 5 つ
- 効果を出すには **個人の習慣を変えるだけでなく、チームのワークフローを再設計する** 必要がある
