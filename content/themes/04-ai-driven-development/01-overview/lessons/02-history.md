# 主要ツールの登場と発展の系譜

AI コーディングツールは、この数年で **「賢い入力補完」から「自律的に働くエージェント」** へと急速に進化してきました。本章では、GitHub Copilot 以前の前史から、Cursor・Claude Code・Codex CLI・Gemini CLI に至る 2026 年までの系譜を、世代ごとの技術的ブレイクスルーとあわせて整理します。

## このページで学べること

- **4 世代の AI コーディングツール** がどう生まれ、何を解決してきたか
- 各世代で起きた **技術的ブレイクスルー**（モデル / UI / エージェント化 / ハーネス）
- GitHub Copilot・Cursor・Claude Code・Codex CLI・Gemini CLI など主要ツールの **立ち位置の違い**
- 「チャット型 / IDE 統合型 / ターミナル型」が **どの順番で登場し、なぜ共存しているか**
- 歴史から読み取る、**次に何が起こりそうか** の勘所

## 全体マップ

細部に入る前に、4 世代の大枠を押さえておきます。

| 世代 | 時期 | 代表的なツール | 支援の粒度 | 主な技術的ブレイクスルー |
| --- | --- | --- | --- | --- |
| 第 0 世代：静的補完 | 〜 2020 年 | IntelliSense / Kite / TabNine（初期） | トークン・行 | 静的解析・言語サーバー |
| 第 1 世代：AI 補完 | 2021 〜 2022 年 | GitHub Copilot / Amazon CodeWhisperer | 行・関数 | Codex / 大規模コード事前学習 |
| 第 2 世代：チャット協働 | 2022 〜 2023 年 | ChatGPT / Claude / Copilot Chat | ファイル・モジュール | 対話型 LLM / コードを貼って相談する UX |
| 第 3 世代：IDE 統合エージェント | 2023 〜 2024 年 | Cursor / Windsurf / Aider | ファイル横断 | リポジトリ全体の RAG / マルチファイル編集 |
| 第 4 世代：ターミナル型自律エージェント | 2024 〜 2025 年 | Claude Code / Codex CLI / Gemini CLI | 機能・タスク単位 | ツール使用（Tool Use）・長期自律実行 |
| 第 5 世代：ハーネスエンジニアリング | 2026 年 〜 | MCP / Subagents / Skills / Hooks | プロジェクト全体 | AI の動作環境そのものの設計 |

以下、各世代で何が起きたのかを時系列で追っていきます。

## 第 0 世代：静的解析ベースの補完（〜 2020 年）

AI コーディング以前の時代は、**静的解析と辞書** が主役でした。エディタは構文木やシンボルテーブルを解析し、「この位置で呼べるメソッド」を候補として列挙するだけで、**「次に何を書きたいか」を推測する** ことはできませんでした。

- **IntelliSense（Microsoft）** — Visual Studio に搭載された古典的な補完。型情報ベースの候補表示
- **Kite（2014 〜 2022）** — Python を中心に機械学習ベースの補完を提供。2022 年 11 月に事業終了
- **TabNine（初期）** — ローカルで動く補完エンジン。2019 年に GPT-2 を統合し、LLM 補完の先駆けの一つに

この時代の補完は「タイピング省略」にとどまり、コードの **意図や設計** にまで踏み込むことはありませんでした。

> [!NOTE]
> 後の世代がどれほど進化しても、静的解析の成果は死んでいません。LSP（Language Server Protocol）は現代の AI コーディングツールでも **型情報・定義ジャンプ・エラー検知** の基盤として生き続けています。

## 第 1 世代：AI 補完の登場（2021 〜 2022 年）

### GitHub Copilot の衝撃（2021 年 6 月）

2021 年 6 月、**GitHub Copilot** がテクニカルプレビューとして登場しました。OpenAI の **Codex モデル**（GPT-3 をコードで追加学習したもの）を背景に、周辺コードの文脈を読み取って「次に書きそうな数行」を提案するという体験は、当時としては衝撃的でした。

- **粒度が変わった** — 単語やトークンではなく、**関数単位のコード片** を一気に提示できるようになった
- **意図を読み取り始めた** — コメントに日本語や英語で意図を書くと、それを実装するコードが出てくる
- **ライセンス論争** — 公開リポジトリのコードで学習していることから、ライセンス・著作権の議論が一気に活発化

2022 年 6 月には GA（正式版）がリリースされ、有料サービスとして一般提供が始まりました。

### 競合の登場

Copilot の成功を追って、各社が類似サービスを投入します。

- **Amazon CodeWhisperer**（2022 年プレビュー、後に Amazon Q Developer へ統合）
- **Tabnine（AI 強化版）** — 自社モデル + ローカル実行を強みに再定位
- **Codeium**（当時は補完に特化、後の **Windsurf** につながる）

> [!NOTE]
> 第 1 世代の AI 補完は、あくまで **「エディタ内の局所的な提案」** にとどまっていました。ファイル横断の編集や、テストの実行といった「開発プロセスへの参加」は、まだ先の世代を待つ必要がありました。

## 第 2 世代：チャット協働の時代（2022 〜 2023 年）

### ChatGPT の登場（2022 年 11 月）

2022 年 11 月、**ChatGPT** が公開されると、「AI とコードについて対話する」という全く新しいワークフローが一気に広まりました。エディタに統合されていないにもかかわらず、開発者は **コードを貼って質問 → 回答をコピペ** という半手動の運用で、設計相談・バグ調査・リファクタ提案に AI を使い始めます。

### Claude の参入と長文コンテキスト

2023 年 3 月に **Anthropic Claude** が登場し、同年 7 月の Claude 2 で **10 万トークン級の長大なコンテキストウィンドウ** を武器に、「ファイル全体やドキュメントをまるごと貼って相談する」体験を後押ししました。

### Copilot Chat — 補完から対話へ

2023 年には **GitHub Copilot Chat** がリリースされ、補完ツールだった Copilot にも「対話」のレイヤーが追加されます。ここで **チャット型 UI は補助ではなく主役の一つ** として定着しました。

### 第 2 世代の意義

- **UI のブレイクスルー** — 自然言語での往復対話が標準になった
- **粒度の拡張** — 「1 行単位」から「ファイル単位・モジュール単位」の議論ができるようになった
- **ただし手作業は残った** — コードを貼る・結果を戻すのは依然として人間の役目

> [!TIP]
> 第 2 世代の UX は、現在でもチャット型ツール（Claude / ChatGPT / Gemini）として生き続けています。「導入が簡単・どの言語でも相談できる」という強みは、第 4 世代以降でも失われていません。

## 第 3 世代：IDE 統合エージェント（2023 〜 2024 年）

第 2 世代の課題は明確でした。**「コードを貼って戻す」手作業は、結局開発者のボトルネックになる**。これを解消したのが、エディタそのものに AI を埋め込み、プロジェクト全体を文脈として扱える **IDE 統合型エージェント** です。

### Cursor の台頭（2023 年〜）

**Cursor** は VS Code フォークとして登場し、以下の点で体験を一段引き上げました。

- **リポジトリ全体のインデックス化** — 開いていないファイルも含めて検索・参照できる
- **マルチファイル編集** — 「この機能を追加」という指示から、複数ファイルに同時に変更を加える
- **Composer / Agent モード** — 単発の補完ではなく、タスクを段階的に進めるエージェント的な操作

### Windsurf・Aider などの拡張

- **Windsurf（Codeium 発）** — 補完特化から IDE 統合エージェントへと進化。**Cascade** と呼ばれるエージェント UI を提供
- **Aider** — ターミナルから動くが Git 連携で IDE と協働する、オープンソースの先駆け
- **JetBrains AI Assistant / Visual Studio IntelliCode** — 既存 IDE ベンダーの追随

### 第 3 世代のブレイクスルー

- **プロジェクト全体の文脈把握** — ベクトル検索 / RAG により、リポジトリを「知っている」状態を作れるようになった
- **マルチファイル編集の標準化** — 「単一ファイル補完」から「機能単位のパッチ」へ
- **対話と編集の往復** — 計画 → 編集 → 差分レビュー → 再指示、というループを IDE 内で完結できるようになった

> [!NOTE]
> 2024 年後半には、Anthropic が **Claude 3.5 Sonnet**、OpenAI が **GPT-4 系**、Google が **Gemini 1.5** と、いずれも長文コンテキスト・高コード性能のモデルを揃え、**「どのツールでも裏側のモデルが十分賢い」** 状態が整います。これが第 4 世代を準備しました。

## 第 4 世代：ターミナル型自律エージェント（2024 〜 2025 年）

IDE 統合で「ファイル横断の編集」はできるようになりましたが、**シェルを叩く・テストを走らせる・Git を操作する** といった作業は依然として人間の仕事でした。これを AI に委ね、**ターミナル上で自律的にタスクをこなす** のが第 4 世代です。

### Claude Code（2025 年 2 月）

**Claude Code** は Anthropic 公式のターミナル型エージェントとして 2025 年 2 月にリリースされました。

- **ツール使用（Tool Use / Function Calling）** を全面活用し、ファイル読み書き・シェル実行・Web 取得を AI 自身が判断して行う
- **`CLAUDE.md`** による前提情報の注入、スラッシュコマンド、**Hooks**、**Subagents**、**Skills** など、環境設計の概念を段階的に整備
- **長時間の自律実行** — 「バグを直してテストを通すまで」といった複数ステップのタスクを一気に進められる

### OpenAI Codex CLI（2025 年）

**OpenAI Codex CLI** は、かつて Copilot の基盤だった「Codex」の名を冠して **ターミナル型エージェント** として再登場しました。OpenAI のモデルを中核に、同様にファイル操作・シェル実行・Git 連携を自律で行います。

### Gemini CLI（2025 年）

**Gemini CLI** は Google が提供するターミナル型エージェントで、Gemini モデルの **超長文コンテキスト** と **Google Cloud / Workspace 連携** を強みに据えました。

### Aider とオープンソース勢

**Aider** は第 3 世代の後半から存在していましたが、第 4 世代のトレンドに合わせて Git ネイティブなエージェントとして再評価されます。OSS ゆえに複数モデル（Claude・GPT・Gemini・ローカル LLM）を切り替えられる柔軟さが強みです。

### 第 4 世代のブレイクスルー

- **ツール使用（Tool Use）** — 「何をすべきか」を考えるだけでなく、**手段（ツール）を AI 自身が選んで実行** できるようになった
- **長期自律実行** — 数十ステップに及ぶタスクを、人間の介入を最小限に進められる
- **シェル・Git・FS をネイティブに扱う** — エディタの制約から解放され、「開発環境全体」に AI が乗り込む

> [!CAUTION]
> 自律実行は **強力であると同時に危険** でもあります。ファイル削除・force push・外部 API 呼び出しなど、取り返しがつかない操作をする力を AI に渡すことになるため、**権限制御とレビューのガードレール** が第 5 世代（ハーネスエンジニアリング）で不可欠なテーマになります。

### 3 形態の共存

重要なのは、**第 4 世代が登場したあとも、チャット型・IDE 統合型・ターミナル型が共存している** ことです。新しい世代が古い世代を置き換えるのではなく、用途によって使い分ける形に落ち着きました。

- **チャット型（第 2 世代の末裔）** — 気軽な相談、学習、ブレスト
- **IDE 統合型（第 3 世代）** — 日常の実装・小〜中規模の編集
- **ターミナル型（第 4 世代）** — リファクタ・機能追加・複数ファイルにまたがる改修

## 第 5 世代：ハーネスエンジニアリング（2026 年 〜）

2026 年現在、主戦場は **モデルそのもの** から **AI の動作環境（ハーネス）を設計すること** へと移っています。モデル性能は各社とも横並びに近づき、**差がつくのは「AI にどんな前提・ツール・権限・ワークフローを与えるか」** になってきました。

ハーネスエンジニアリングの主要な構成要素は以下の通りです。

- **MCP（Model Context Protocol）** — AI に外部サービス・DB・ブラウザ操作などの **ツール拡張** を与える標準プロトコル
- **Subagents** — タスクごとに独立したコンテキストでエージェントを走らせ、並列化・役割分担する
- **Skills / スラッシュコマンド** — 定型ワークフローをコマンド化し、再利用可能な手順として蓄積する
- **Hooks** — AI の動作の前後にフォーマット・テスト・通知などを差し込み、ガードレールを敷く
- **権限制御** — 自律実行の範囲を明示的に設計し、安全と速度を両立させる

> [!TIP]
> 第 5 世代は、「どのツールを使うか」よりも **「どう設計して使うか」** が生産性を分ける時代です。詳細は [コンテキストエンジニアリング](/themes/04-ai-driven-development/01-overview/context-engineering) を参照してください。

## 世代ごとのブレイクスルーまとめ

各世代を「何が新しくなったか」の観点で並べ直すと、進化の方向性がよく見えます。

| 世代 | 新しくなったもの | 人間が AI に渡すもの | AI が人間に返すもの |
| --- | --- | --- | --- |
| 第 0 世代 | 静的解析 | 書いている途中のコード | 候補一覧 |
| 第 1 世代 | 事前学習されたコードモデル | 周辺コード＋コメント | 次の数行 |
| 第 2 世代 | 対話型 LLM | 自然言語＋貼り付けコード | 説明・修正提案 |
| 第 3 世代 | プロジェクト全体の RAG | 意図＋リポジトリ | マルチファイルの差分 |
| 第 4 世代 | ツール使用と自律実行 | 高レベルな目的 | コミット可能な成果物 |
| 第 5 世代 | ハーネス（環境設計） | 設計されたワークフロー | 安全かつ再現性ある成果物 |

傾向として、**「人間が渡す情報はより抽象的に」「AI が返す成果物はより具体的で完成度が高く」** なってきています。

## 歴史から読み取れること

### 1. ボトルネックは「コード生成」から「制御と評価」へ移ってきた

初期は「いかに賢く書かせるか」が課題でしたが、第 4 世代以降は **「自律に任せた結果をどうレビュー・制御するか」** のほうが難しくなっています。CI・テスト・Hooks・レビュー文化の重要性は相対的に高まっています。

### 2. 世代を超えて「負けない」強みは UX と配布形態

GitHub Copilot が強いのは GitHub に、Cursor が強いのはエディタに、Claude Code が強いのはシェルに、**それぞれ「開発者が元々いる場所」に滑り込んだから** です。モデル性能は時間とともに横並びになりますが、**どこで動くか** の設計は長く効きます。

### 3. 3 形態は当面共存する

チャット型・IDE 統合型・ターミナル型は **用途ごとに最適解が違う** ため、どれかが消えるよりも、開発者側が **3 つを切り替えて使うスキル** を身につける方向で収束しています。

## まとめ

- AI コーディングは、**静的補完 → AI 補完 → チャット協働 → IDE 統合 → ターミナル自律 → ハーネス** と 5 つの世代を経て進化してきた
- 第 1 世代（Copilot）の価値は **「次の数行を書く」**、第 4 世代（Claude Code / Codex CLI / Gemini CLI）の価値は **「タスクを自律的に完了させる」** に進化した
- 新世代が古い世代を置き換えるのではなく、**チャット / IDE / ターミナルの 3 形態が共存** する形で定着した
- 2026 年現在は **ハーネスエンジニアリング** が主戦場で、モデル選びよりも **AI の動作環境の設計** が成果を分ける
- 歴史は「人間の渡すものは抽象化し、AI の返すものは具体化する」方向に進んでいる。**評価・レビュー・設計** の重要性はむしろ上がっている
