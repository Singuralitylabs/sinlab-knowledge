# コンテキストエンジニアリング

Claude Code・Cursor・Codex などの AI コーディングエージェントの潜在能力を最大限に引き出す概念が **コンテキストエンジニアリング** です。AI の「記憶領域」であるコンテキストウィンドウの仕組みを理解し、適切に管理することで、コーディングの精度を何倍にも向上させることができます。

## このページで学べること

- **コンテキストウィンドウ** の仕組みと、AI の「忘れっぽさ」が発生する理由
- LangChain が提唱する **記述（Write）/ 選択（Select）/ 圧縮（Compress）/ 分離（Isolate）** の 4 原則
- 中・大規模プロジェクトで精度を維持する **仕様書駆動開発（スペック駆動開発）** のワークフロー
- `/init` `/context` `/compact` `/agents` など、コンテキスト管理に関わる主要コマンドの役割

## コンテキストウィンドウの基礎

AI コーディングエージェント（LLM）には **コンテキストウィンドウ** と呼ばれる記憶領域が設定されており、これが AI のパフォーマンスを左右する決定的な要因となります。

![コンテキストウィンドウの構成と記憶の限界](/images/ai-driven-development/context-engineering/context-memory.png)

### 記憶の限界

人間の短期記憶と同様に、LLM が一度に保持できる情報量には上限があります。たとえば Claude Code では通常 20 万トークン程度が上限です。

### コンテキスト溢れの問題

指示や対話が重なり、ウィンドウの限界を超えると、過去の履歴から順に「記憶」が抜け落ちていきます。これにより、以下のような現象が発生します。

- AI が以前の指示を無視する
- 禁止したはずの実装を繰り返す
- プロジェクトの規約や前提条件を忘れる

### コンテキストウィンドウの構成要素

コンテキストウィンドウには、以下のデータがすべて含まれます。

- ユーザーからのインプット（プロンプト）
- AI によるツール実行（ファイル検索、コード編集など）の履歴
- 最終的なアウトプット（メッセージ）

> [!NOTE]
> ツール実行の結果（ファイルの中身やコマンドの出力）もすべてコンテキストに蓄積されます。大きなファイルを何度も読ませると、あっという間にウィンドウが埋まってしまうため注意が必要です。

## コンテキストエンジニアリングの 4 原則

効率的な開発環境を構築するために、以下の 4 つのアプローチを組み合わせてコンテキストを最適化します。

![コンテキストエンジニアリングの4原則](/images/ai-driven-development/context-engineering/four_context_engineering.png)

### 記述（Write）

AI の「忘れっぽさ」を補うため、**外部メモリとしてドキュメントを残す** 原則です。

`docs/` ディレクトリを作成し、要件定義書、仕様書、タスク管理用のチケットファイル、守ってほしいルールなどをマークダウン形式で管理します。コンテキストウィンドウから過去の会話が消えても、AI はこれらのファイルを参照することで常に最新の要件を把握できます。

### 選択（Select）

**必要なタイミングで、適切なファイルのみを AI に読み込ませる** 技術です。

出力精度が低下した際や、特定のフェーズに移行する際に、関連するドキュメントを明示的に参照させます。すべてのドキュメントを毎回読み込ませるのではなく、「今このタスクで必要な情報だけ」を渡すことが重要です。

### 圧縮（Compress）

限られたコンテキスト容量を節約するため、**情報を要約する** 原則です。

- **オートコンパクト機能** — Claude Code などのツールには、コンテキストが少なくなった際に自動で会話履歴を圧縮する機能が備わっています
- **手動圧縮** — `/compact` などのコマンドを使用して、会話履歴をクリアしつつ、その内容の要約のみを保持することで容量を確保できます

### 分離（Isolate）

メインのコンテキストを汚染しないよう、**タスクごとにエージェントやコンテキストを分ける** 原則です。

- **サブエージェントの活用** — デザイン調査、パフォーマンス改善、セキュリティ監査など、特定のタスク専用のサブエージェントを立ち上げ、独立したコンテキストで作業させます
- **要約のフィードバック** — サブエージェントでの成果を要約し、必要な情報だけをメインのコンテキストに戻すことで、メインウィンドウの圧迫を防ぎます

> [!TIP]
> 4 原則は独立したテクニックではなく、組み合わせて効果を発揮します。「外部ドキュメント（Write）を、必要なときだけ参照（Select）させ、長くなった会話は圧縮（Compress）し、重い調査はサブエージェントに分離（Isolate）する」—— この流れを常に意識しましょう。

## 仕様書駆動開発（スペック駆動開発）の実践

中・大規模プロジェクトや複雑な個人開発において、精度の高い開発を行うためのワークフローです。ここでは Claude Code を例に説明しますが、Cursor や Codex など他のツールでも基本的な考え方は同じです。

ポイントは **ドキュメントを人間が手書きで準備しない** ことです。漠然としたアイデアの段階から AI と対話しながら `docs/` を育て、それを外部メモリ（Write）として実装フェーズで参照させます。

### 開発サイクル

1. **計画（`/plan` モードでの対話）** — いきなりコードを書かせず、まずは Claude Code の **Plan モード** や `/plan` 相当の計画モードで、作りたいものを AI と一緒に壁打ちします。機能要件・非機能要件・技術選定・懸念点を質問で引き出してもらい、方針を固めます
2. **ドキュメントのドラフト生成** — 計画の合意が取れたら、AI に `docs/` 配下のファイル群をドラフトさせます。ゼロから人間が書くのではなく、**1 の会話内容を元に AI が初稿を作る** のが基本です
3. **レビューと加筆** — 生成されたドラフトを人間がレビューし、ドメイン知識・制約・好みを補記します。ここで初めて「外部記憶」として信頼できる仕様書になります
4. **初期化** — `/init` などでプロジェクトをスキャンさせ、`CLAUDE.md` と `docs/` を AI が常に参照できる状態にします
5. **フェーズ別実装** — `tasks.md` のチェックボックスに従い、フェーズごとに実装を依頼します。1 フェーズが終わるたびにコンテキストをリフレッシュ（`/clear` や `/compact`）して、該当フェーズの docs のみを読み直させます
6. **進捗同期** — タスク完了ごとに `tasks.md` や関連ドキュメントを更新します。これも AI に更新させ、人間は差分をレビューするだけに留めます

### docs/ に揃えたい成果物

上記 2〜3 のステップを経て、最終的に `docs/` 配下には以下のようなファイルが揃います。いずれも **AI が初稿を書き、人間がレビューで仕上げる** 前提です。

| ファイル | 役割 | 主な生成タイミング |
| --- | --- | --- |
| `requirements.md` | 要件定義。アプリの目的・ユーザー像・主要機能 | 計画フェーズの直後 |
| `architecture.md` | 技術スタック、レイヤー構成、ディレクトリ構造の方針 | 計画フェーズ〜初期実装前 |
| `data_structure.md` | DB 設計・主要データ型・エンティティ関係 | アーキテクチャ確定後 |
| `tasks.md` | 実装タスクのリストと進捗（チェックボックス形式） | 要件・設計確定後、実装開始前 |
| `CLAUDE.md` | コーディング規約、AI への定型的な指示 | `/init` で自動生成 → 手動で調整 |

> [!TIP]
> `/plan` モードは「まず計画を立ててから実装に入る」ための安全装置です。AI がいきなりファイルを書き換える前に、方針の合意をとる場として活用しましょう。計画段階で docs の骨子まで詰めておくと、そのまま `requirements.md` / `tasks.md` のドラフトに転用できます。

> [!IMPORTANT]
> 「ドキュメントを書く手間が増える」と感じるかもしれませんが、初稿は AI に書かせるので手間の大半は削れます。人間の仕事は **「AI の書いた仕様書に、AI が知らない前提を足す」** ことに集中し、以降の実装で何度でも参照できる外部メモリに育てていきましょう。

## コンテキストに関わる主要コマンド

AI コーディングエージェントを使いこなすための主要な機能を紹介します。ここでは Claude Code のコマンドを例に挙げますが、他のツールにも同様の機能が用意されていることが多いです。

### /init

プロジェクトの構造を解析し、`CLAUDE.md` ファイルを自動生成します。AI がプロジェクト全体を把握するための初期設定として、**開発の最初に実行** することを推奨します。

### /context

現在のコンテキストウィンドウの使用状況を可視化します。どのリソース（メッセージ、システムプロンプト、ツール出力など）が領域を圧迫しているかを把握できます。

### /compact

会話履歴を要約して圧縮し、コンテキストの空き容量を確保します。Claude Code では **会話がある程度進んでコンテキストの残量が少なくなると自動的に `/compact` が実行される**（オートコンパクト）ため、通常は意識しなくても圧縮が走ります。

一方で、自動実行のタイミングが来る前に **話題を切り替えたいとき** や、**残しておきたい情報を指定して圧縮したいとき** は手動での実行が有効です。`/compact TypeScript の型設計に関する議論を優先して残して` のようにフォーカスを指示すると、要約後も重要情報が失われにくくなります。

> [!TIP]
> オートコンパクトは便利ですが、実行タイミングは AI 任せです。区切りの良いところで手動 `/compact` しておくと、要約される内容を自分でコントロールできます。

### /agents

新しいサブエージェントを作成し、特定の役割（例：リサーチャー、レビュアー）を割り当てます。メインのコンテキストを汚さずに、独立した調査や作業を行わせることができます。

## コンテキストに関わるその他の機能

### CLAUDE.md（ルールファイル）

プロジェクトのルートに配置する設定ファイルです。コーディング規約やプロジェクト固有の指示を記述しておくと、会話を開始するたびに自動でコンテキストに読み込まれます。**記述（Write）** の原則を実現する中核的な仕組みです。

### メモリ機能

AI が会話を通じて学んだプロジェクトの知見を、セッションをまたいで保持する機能です。「この設定を覚えておいて」と指示することで、次回以降の会話でも同じ前提で作業を続けられます。

### カスタムスラッシュコマンド（Skills）

よく使うプロンプトをスラッシュコマンドとして登録できる機能です。コミットメッセージの作成やコードレビューの手順を定型化しておくことで、毎回同じ指示を入力する必要がなくなり、コンテキストの消費を抑えられます。

### @メンション（ファイル参照）

会話中に `@ファイル名` と入力することで、特定のファイルを明示的にコンテキストに読み込ませることができます。**選択（Select）** の原則を手軽に実践する方法です。

> [!TIP]
> `@` によるファイル参照は、プロジェクト全体を読み込ませるより圧倒的に効率的です。「このファイルを見て修正して」と `@` で指定する習慣をつけましょう。

## まとめ

コンテキストエンジニアリングは、単なるプロンプトエンジニアリングを超えた、**AI との協業における「管理技術」** です。特に、小規模なアプリ（ToDo アプリ等）を超えて、データベース・認証・決済などを含む複雑なアプリケーションを構築する場合、この概念の理解と実践が不可欠となります。

ドキュメントを通じて AI に「外部記憶」を与え、サブエージェントによって「思考の分離」を実現することで、大規模なプロジェクトにおいても一貫性のある高品質なコード生成が可能になります。
