# メモリとコンテキスト管理

CLAUDE.md によるプロジェクト知識の永続化、自動メモリ、コンテキスト圧縮、セッション管理など、Claude Code がどのように情報を記憶・活用するかを詳しく解説します。

## このページで学べること

- **CLAUDE.md** の役割と、プロジェクトルールを Claude Code に伝える方法
- グローバル・プロジェクト・サブディレクトリの **3 階層構造** と `@import` の使い方
- **自動メモリ** による学習パターンの永続化の仕組み
- **`/compact`** コマンドによるコンテキスト圧縮テクニック
- **セッション管理**（再開・一覧・リネーム）のワークフロー
- **タスクリスト** による TODO 追跡と進捗管理

## CLAUDE.md とは

`CLAUDE.md` は、Claude Code がセッション開始時に最初に読み込むマークダウンファイルです。プロジェクトの概要、ビルドコマンド、コーディング規約、アーキテクチャの方針など「Claude に常に覚えていてほしい情報」をここに書いておきます。

人間にとっての「プロジェクト README + チーム開発ルール集」にあたるものを、AI 向けに最適化した形式で記述するイメージです。Claude Code はこのファイルの内容をシステムプロンプトの一部として扱い、すべてのやり取りに反映します。

```markdown
# CLAUDE.md

## Project Overview
Next.js 16 (App Router) + TypeScript + Tailwind CSS 4 のWebサイト。
日本語コンテンツの教育用サイト。

## Commands
npm run dev      # 開発サーバー起動 (localhost:3000)
npm run build    # プロダクションビルド
npm run lint     # ESLint 実行

## Architecture
- App Router: app/ ディレクトリベースルーティング
- Server Components がデフォルト
- Client Component は最小限（状態管理が必要な場合のみ）

## Conventions
- TypeScript strict mode を使用
- コンテンツは全て日本語
- コンポーネントは app/components/ に配置
- パスエイリアス: @/* → プロジェクトルート
```

> [!TIP]
> CLAUDE.md は 500 行以下に抑えるのがベストプラクティスです。長くなりすぎるとコンテキストウィンドウを圧迫します。詳細な手順やリファレンスは Skills に分離しましょう。

### CLAUDE.md に書くべきこと / 書かないこと

| 書くべきこと | 書かないこと |
| --- | --- |
| プロジェクトの概要と目的 | API キーや秘密情報 |
| ビルド / テスト / デプロイコマンド | 長大なコードスニペット |
| コーディング規約とスタイルルール | 頻繁に変わる一時的な情報 |
| アーキテクチャの方針 | 全ファイルの詳細な説明 |
| 「常にやること / やらないこと」ルール | 汎用的な知識（Claude が既に知っていること） |
| ディレクトリ構造の概要 | — |

> [!CAUTION]
> API キー・トークン・パスワードなどの秘密情報は絶対に CLAUDE.md に書かないでください。CLAUDE.md はリポジトリにコミットされるため、共有リポジトリで情報漏洩の原因になります。

## CLAUDE.md の階層構造

CLAUDE.md は 3 つの階層に配置できます。Claude Code はセッション開始時にこれらをすべて読み込み、内容をマージします。より限定的なスコープの指示が優先されます。

### 読み込み階層

| レベル | 配置場所 | 役割 |
| --- | --- | --- |
| Level 1 - グローバル | `~/.claude/CLAUDE.md` | 全プロジェクト共通のルール（個人設定、コーディングスタイル） |
| Level 2 - プロジェクト | `/project-root/CLAUDE.md` | プロジェクト固有のルール（アーキテクチャ、コマンド、規約） |
| Level 3 - サブディレクトリ | `/project-root/src/CLAUDE.md` | ディレクトリ固有のルール（そのフォルダの作業時のみ読み込み） |

優先度：**サブディレクトリ（最優先） > プロジェクト > グローバル（最下位）**

> [!NOTE]
> サブディレクトリの CLAUDE.md は、Claude Code がそのディレクトリ内のファイルに触れたタイミングで読み込まれます。全体を最初から読み込むわけではないため、巨大モノレポでも負荷を抑えられます。

### @import でファイルを分割する

`@path/to/file.md` 構文を使うと、CLAUDE.md から別のマークダウンファイルをインポートできます。これにより、内容を分割して管理しやすくなります。

```markdown
# CLAUDE.md

## 基本ルール
- TypeScript strict mode を使用
- テストは必ず書く

## 詳細ドキュメント
@docs/architecture.md
@docs/coding-standards.md
@docs/api-guidelines.md
```

> [!IMPORTANT]
> `@import` されたファイルは CLAUDE.md と同様にシステムプロンプトに追加されます。合計サイズがコンテキストウィンドウを圧迫しないよう注意しましょう。個々のファイルも簡潔に保つことが重要です。

## 自動メモリ

Claude Code は作業中に学んだパターンやプロジェクト固有の知識を **自動的に記録** します。明示的に「覚えて」と指示しなくても、会話の中で重要だと判断した情報をメモリに保存し、次のセッションでも活用します。

### 自動メモリに記録される情報の例

| カテゴリ | 具体例 |
| --- | --- |
| コーディングパターン | このプロジェクトでは `fetch` の代わりに `axios` を使う、エラーハンドリングは Result 型で統一する |
| アーキテクチャの決定 | 状態管理は Zustand を使用、API レイヤーは Repository パターンで実装 |
| ユーザーの好み | 日本語でコメントを書く、型は明示的に書く、テストは describe/it スタイルで書く |
| プロジェクト固有の知識 | データベースのテーブル構造、API エンドポイントの規則、特殊な命名規則 |

### メモリの保存場所

自動メモリは以下の場所に保存されます。これらのファイルはユーザーが直接編集することも可能です。

```bash
# ユーザーレベルのメモリ
~/.claude/memories/

# プロジェクトレベルのメモリ
.claude/memories/
```

> [!TIP]
> 自動メモリは便利ですが、重要なルールは必ず CLAUDE.md にも明記しましょう。CLAUDE.md は毎回確実に読み込まれるため、最も信頼性の高い情報伝達手段です。自動メモリは補完的な役割として活用するのがベストです。

## コンテキスト圧縮（/compact）

Claude Code との会話が長くなると、コンテキストウィンドウ（Claude が一度に保持できる情報量）が不足してきます。`/compact` コマンドは、これまでの会話を要約して圧縮し、コンテキストの空きを確保します。

### 処理フロー

`長い会話` → `/compact 実行` → `要約を生成` → `圧縮済み要約`

要約後のコンテキストは元の会話より大幅に小さくなり、新しい会話のための余裕が生まれます。

### フォーカス指示つき /compact

`/compact` にはオプションでフォーカス指示を添えることができます。何を優先的に残すかを指定することで、圧縮後も重要な情報が失われにくくなります。

```bash
# TypeScript の型情報を優先的に残す
/compact TypeScript の型情報を優先的に残して

# アーキテクチャの議論を残す
/compact アーキテクチャに関する決定事項を優先して

# 直近のエラーのデバッグ情報を残す
/compact 直近のバグ修正に関連する情報を残して
```

### /compact を使うタイミング

- **会話が長くなったとき** — 数十回のやり取りを超えたら圧縮を検討します
- **Claude の応答が劣化したとき** — コンテキスト不足で指示を忘れ始めたら圧縮のサインです
- **話題を切り替えるとき** — 新しいタスクに移る前に、前のタスクの情報を圧縮します
- **コンテキストウィンドウの警告が出たとき** — Claude Code がコンテキスト不足を通知した場合

> [!TIP]
> コンテキスト使用量が高くなると、Claude Code は自動的に `/compact` を提案または実行することがあります。定期的に手動で `/compact` するのが効果的です。

## セッション管理

Claude Code は会話の履歴をセッションとして保存しています。過去のセッションを再開したり、名前をつけて管理したりすることで、作業の中断・再開をスムーズに行えます。

### 主なコマンド

| コマンド | 説明 | 使いどころ |
| --- | --- | --- |
| `claude -c` | 直前のセッションを再開 | 作業の中断後に続きを再開したいとき |
| `claude -r` | 過去のセッション一覧から選択して再開 | 特定の過去のセッションに戻りたいとき |
| `/rename` | 現在のセッションに名前をつける | 重要なセッションを後から見つけやすくしたいとき |
| `/clear` | 会話履歴をクリアして新規セッション | まったく新しいタスクに取りかかるとき |

### セッション管理のワークフロー例

1. **朝：新しいセッションを開始** — `claude` で起動し、認証機能の実装を開始
2. **中断：セッションに名前をつける** — `/rename auth-implementation` でセッションに名前をつけてから離席
3. **午後：セッションを再開** — `claude -c` で直前のセッションから再開し、続きの作業
4. **翌日：別のセッションを選択** — `claude -r` で一覧から「auth-implementation」を選択して再開

> [!TIP]
> 重要な作業のセッションには必ず `/rename` で名前をつけておきましょう。セッション名があると `claude -r` で選択するときに、目的のセッションをすぐに見つけられます。

## タスクリスト

Claude Code には TODO 追跡のためのタスクリスト機能が組み込まれています。複雑な作業を分解し、進捗をリアルタイムで管理できます。特筆すべき点として、タスクリストは **コンテキスト圧縮（`/compact`）後も保持される** ため、長いセッションでも作業の全体像を見失いません。

### 操作方法

| 操作 | 説明 |
| --- | --- |
| `Ctrl+T` | タスクリストの表示/非表示を切り替え。作業中にいつでもトグルして進捗を確認できます |
| 自動管理 | Claude が自動的にステータスを更新。未着手 / 進行中 / 完了 のステータスが作業に応じて変わります |

### タスク進行の例

「ユーザー認証を実装して」と依頼した場合のタスクリストの変化です。

**作業開始時:**

```text
● 認証ライブラリの選定と導入      （進行中）
○ ログイン/ログアウト API の実装  （未着手）
○ 認証ミドルウェアの作成          （未着手）
○ テストの作成                    （未着手）
```

**作業途中:**

```text
✓ 認証ライブラリの選定と導入      （完了）
✓ ログイン/ログアウト API の実装  （完了）
● 認証ミドルウェアの作成          （進行中）
○ テストの作成                    （未着手）
```

凡例: `✓` 完了 / `●` 進行中 / `○` 未着手

> [!NOTE]
> **/compact とタスクリストの関係:** コンテキスト圧縮を実行すると会話の詳細は要約されますが、タスクリストはそのまま保持されます。これにより、圧縮後も「どこまで完了したか」「何が残っているか」を正確に把握できます。
