# Claude Code の基本構造

`.claude` ディレクトリの構造、`settings.json` の設定、`CLAUDE.md` の書き方、対話モードと非対話モードの使い分け、権限システムまで、Claude Code の基本操作を一通り解説します。

## このページで学べること

- `.claude` ディレクトリの役割とファイル構成
- `settings.json` による設定管理の仕組み
- `CLAUDE.md` の役割と効果的な書き方
- インストールから起動までの手順
- 対話モードと非対話モードの使い分け
- 権限システムと安全なツール実行の仕組み
- 日常の開発で使う典型的なワークフロー

## .claude ディレクトリの構造

Claude Code の設定やメモリは、**ホームディレクトリ**と**プロジェクトルート**の 2 箇所に配置される `.claude` ディレクトリで管理されます。

### ホームディレクトリ（~/.claude/）

ユーザー全体に適用される設定やルールが格納されます。どのプロジェクトで Claude Code を起動しても、ここの設定は常に読み込まれます。

```text
~/.claude/
├── CLAUDE.md          # グローバルな指示（全プロジェクト共通のルール）
├── settings.json      # ユーザー設定（許可ツール、APIキー設定など）
├── credentials.json   # 認証情報（自動管理、手動編集不要）
├── commands/          # カスタムスラッシュコマンド（グローバル）
│   └── my-command.md
├── memories/          # 自動メモリ（プロジェクト横断で学習した情報）
└── projects/          # プロジェクト別の自動メモリ
    └── <project-hash>/
        └── MEMORY.md
```

### プロジェクトルート（.claude/）

プロジェクト固有の設定が格納されます。チームで共有する場合は Git で管理することを推奨します。

```text
your-project/
├── CLAUDE.md          # プロジェクト固有の指示（最も重要）
├── .claude/
│   ├── settings.json  # プロジェクト設定（許可ツール、MCP設定など）
│   └── commands/      # プロジェクト固有のカスタムコマンド
│       └── deploy.md
└── src/
    └── CLAUDE.md      # サブディレクトリ固有の指示（オプション）
```

### 設定の読み込み優先順位

`settings.json` は複数の場所に置くことができ、スコープが狭い順に優先して適用されます。プロジェクト側で未定義のキーは、より広いスコープの設定にフォールバックします。

| 優先度 | 配置場所 | 用途 |
| --- | --- | --- |
| 1（最優先） | `your-project/.claude/settings.json` | プロジェクト固有の設定（MCP サーバー、許可ツールなど） |
| 2 | `~/.claude/settings.json` | ユーザー全体の設定（グローバルな許可ツール、デフォルト設定） |
| 3 | Claude Code 組み込みのデフォルト | 設定ファイルがない場合に使われる初期値 |

> [!TIP]
> プロジェクトの `.claude/settings.json` は Git にコミットしてチームで共有できます。ただし、API キーや認証情報が含まれていないか必ず確認してください。機密情報は `settings.local.json` など Git 管理外のファイルに分けて置くのが安全です。

## settings.json の設定

`settings.json` は Claude Code の動作を制御する設定ファイルです。許可するツール、MCP サーバーの接続、環境変数などを定義します。

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Write",
      "Edit",
      "Bash(npm run *)",
      "Bash(git *)"
    ],
    "deny": [
      "Bash(rm -rf *)"
    ]
  },
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@anthropic-ai/mcp-playwright"]
    }
  },
  "env": {
    "NODE_ENV": "development"
  }
}
```

### 主な設定項目

| キー | 説明 | スコープ |
| --- | --- | --- |
| `permissions` | ツール実行の許可 / 拒否リスト | ユーザー / プロジェクト |
| `mcpServers` | MCP サーバーの接続設定 | プロジェクト |
| `env` | Claude Code 実行時の環境変数 | ユーザー / プロジェクト |
| `hooks` | イベント駆動のフック設定 | ユーザー / プロジェクト |

> [!NOTE]
> `permissions` のパターンには `Bash(npm run *)` のようにワイルドカードを使えます。これにより「`npm run` で始まるコマンドは全て許可」といった柔軟な設定が可能です。`deny` に挙げたパターンは `allow` より優先されるため、`rm -rf` のような危険コマンドを確実に弾けます。

## CLAUDE.md — プロジェクトの "取扱説明書"

`CLAUDE.md` は、Claude Code に対する**永続的な指示書**です。プロジェクトのルール、コーディング規約、アーキテクチャの方針などを記述しておくと、セッションを開始するたびに Claude が自動的に読み込み、それに従って動作します。

### CLAUDE.md とは

`CLAUDE.md` はいわば「Claude Code へのシステムプロンプト」です。毎回チャットで伝え直す必要なく、プロジェクト固有の知識やルールを常に反映させることができます。

- **永続的な指示**: セッションごとに自動で読み込まれる
- **チームで共有**: Git 管理して全員に同じルールを適用できる
- **精度向上**: プロジェクト文脈を Claude が正しく理解し、的外れなコードを減らせる

### 配置場所と読み込み順序

`CLAUDE.md` は複数の場所に配置でき、すべてが統合されて読み込まれます。スコープが狭いほど優先されるため、サブディレクトリの指示がプロジェクト全体の指示を上書きします。

1. **`~/.claude/CLAUDE.md`（グローバル）**
   全プロジェクト共通のルール。コーディングスタイルの好み、使用言語の設定など、個人的な指示に最適です。
2. **`your-project/CLAUDE.md`（プロジェクト）**
   プロジェクト固有のルール。アーキテクチャ、技術スタック、ビルドコマンドなどを記述します。**最も重要な CLAUDE.md** で、Git で管理してチームに共有しましょう。
3. **`your-project/src/CLAUDE.md`（サブディレクトリ）**
   特定のディレクトリに限定した指示。そのディレクトリ内のファイルを操作する際に追加で読み込まれます。

> [!IMPORTANT]
> プロジェクトの `CLAUDE.md` は、新しいメンバーの「オンボーディングドキュメント」に近い存在です。Claude だけでなく人間の開発者にとっても有用な情報を書いておくと、ドキュメントの二重管理を避けられます。

### 何を書くべきか

`CLAUDE.md` に書く内容に正解はありませんが、以下のカテゴリが特に効果的です。

- **よく使うコマンド**: ビルド、テスト、lint、デプロイなどのコマンドを記述。Claude がコマンドを推測する必要がなくなります。
- **アーキテクチャと技術スタック**: フレームワーク、ディレクトリ構造、主要なパターンを説明。新しいコードが既存のアーキテクチャに沿って生成されます。
- **コーディング規約**: 命名規則、インポート順序、エラーハンドリングのルールなど。チームのスタイルに一致したコードが生成されます。
- **やってはいけないこと（禁止事項）**: 「any 型を使わない」「console.log をコミットしない」など、Claude に避けさせたいパターンを明記します。

### CLAUDE.md の実例

プロジェクトルートに置く `CLAUDE.md` の具体的な書き方の例です。`/init` コマンドで自動生成することもできます。

````markdown
# CLAUDE.md

## プロジェクト概要
ECサイトのバックエンドAPI。Next.js App Router + Prisma + PostgreSQL。

## コマンド
```bash
npm run dev       # 開発サーバー起動 (localhost:3000)
npm run build     # プロダクションビルド
npm run test      # Jest でテスト実行
npm run lint      # ESLint + Prettier
npm run db:push   # Prisma スキーマを DB に反映
```

## アーキテクチャ
- app/ ディレクトリベースルーティング（App Router）
- app/api/ に Route Handlers を配置
- lib/ に共有ユーティリティ
- prisma/schema.prisma でデータモデル定義

## コーディング規約
- TypeScript strict モード、any 型禁止
- コンポーネントは named export を使用
- エラーハンドリングは Result 型パターンで統一
- テストは __tests__/ ディレクトリに配置

## 禁止事項
- console.log をコミットしない（logger を使う）
- API キーをハードコードしない（環境変数を使う）
- default export は page.tsx と layout.tsx のみ
````

### /init で自動生成する

対話モードで `/init` コマンドを実行すると、Claude がプロジェクトの構造を分析して `CLAUDE.md` を自動生成してくれます。これをベースに必要な情報を加筆するのが効率的です。

```bash
$ claude
> /init

Claude: プロジェクトを分析しています...

✓ package.json を解析 → Next.js 16 + TypeScript プロジェクト
✓ ディレクトリ構造を分析 → App Router パターン
✓ 設定ファイルを確認 → ESLint, Tailwind CSS, PostCSS

CLAUDE.md を生成しました。内容を確認してください。
```

> [!TIP]
> `CLAUDE.md` は Markdown 形式で書きます。箇条書きやコードブロックを活用して、Claude が解析しやすい構造にしましょう。長くなりすぎるとコンテキストを圧迫するため、簡潔に書くことが重要です。

## 基本的な使い方

### インストールと初期設定

Claude Code は npm パッケージとして配布されています。Node.js 18 以上の環境で、以下のコマンドを実行するだけでセットアップが完了します。

```bash
# グローバルインストール
npm install -g @anthropic-ai/claude-code

# プロジェクトディレクトリで起動
cd your-project
claude

# 初回は認証フローが開始
# ブラウザで Anthropic アカウントにログイン → 認証完了
```

セットアップは次のような流れになります。

1. `npm install -g @anthropic-ai/claude-code` で CLI をインストール
2. プロジェクトディレクトリで `claude` を実行して起動
3. ブラウザで Anthropic アカウントを認証
4. 対話モードに入ったら `/init` を実行して `CLAUDE.md` を自動生成

`/init` を実行することで、プロジェクトの文脈が最初から Claude に伝わり、以降のやりとりが的確になります。

Claude Code には大きく分けて**対話モード**と**非対話モード**の 2 つの使い方があります。状況に応じて使い分けることで、効率的に開発を進められます。

### 対話モード

最も基本的な使い方です。ターミナルで `claude` と入力して起動し、自然言語で指示を出します。Claude は質問に答えたり、コードを編集したり、コマンドを実行したりします。

```text
$ claude
╭──────────────────────────────────────╮
│ Claude Code                          │
│                                      │
│ /help for available commands         │
╰──────────────────────────────────────╯

> このプロジェクトの構造を教えて

Claude: このプロジェクトは Next.js App Router を使った
Web アプリケーションです。主な構造は...

> src/components/Header.tsx にダークモード切替ボタンを追加して

Claude: Header.tsx を編集してダークモード切替ボタンを
追加しました。変更内容は...
```

### 非対話モード（-p / --print）

ワンショットで質問に答えてもらいたい場合や、スクリプト・CI パイプラインに組み込みたい場合に使います。結果を出力して即座に終了します。

```bash
# 質問に回答して終了
claude -p "package.json の依存関係を一覧にして"

# パイプと組み合わせて使う
cat error.log | claude -p "このエラーログの原因を分析して"

# 特定のファイルを渡して分析
claude -p "このコードのセキュリティリスクを指摘して" < auth.ts
```

### 初期プロンプト付き起動

対話モードで起動しつつ、最初の質問を一緒に渡すことができます。また、過去のセッションを引き継ぐオプションも便利です。

```bash
# 質問を渡して対話モードを開始
claude "このプロジェクトのテストカバレッジを改善したい"

# セッション再開
claude -c              # 直前のセッションを再開
claude -r              # 過去のセッション一覧から選択
```

> [!TIP]
> まずは対話モードで使い始めるのがおすすめです。慣れてきたら非対話モード（`-p`）を CI やスクリプトに組み込むと、自動化の幅が広がります。たとえば「PR に対して自動で lint コメントを付ける」「エラーログを毎朝要約する」といったジョブを Claude に任せられます。

## 権限システム

Claude Code はファイルの編集やコマンドの実行を行う際に、ユーザーの**明示的な許可**を求めます。これにより、意図しない変更やコマンド実行を防ぎ、安全に作業を進められます。

### 権限確認の種類

ツールの種類ごとに、初期状態での挙動が異なります。

| 種類 | 操作 | デフォルトの挙動 |
| --- | --- | --- |
| 読み取り（Read） | ファイルの内容を読む | 自動で許可される |
| 書き込み（Write / Edit） | ファイルの作成・編集・削除 | 操作ごとにユーザー確認。diff が表示される |
| コマンド実行（Bash） | シェルコマンドの実行 | 実行前にコマンド内容を表示し、承認まで実行されない |

- **読み取り（Read）**: ファイル内容の参照は副作用がないため、デフォルトで許可されています。
- **書き込み（Write / Edit）**: ファイルの作成・編集・削除は毎回確認が入り、変更前後の diff が表示されます。意図しない書き換えを承認前に検知できます。
- **コマンド実行（Bash）**: シェルコマンドはもっとも影響範囲が大きいため、コマンド文字列を確認してから承認します。

### 権限の事前設定

毎回の確認が煩わしい場合は、`settings.json` で許可ルールを事前に設定できます。よく使うコマンドやツールを許可リストに追加しておくと、スムーズに作業できます。

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Edit",
      "Write",
      "Bash(npm run dev)",
      "Bash(npm run build)",
      "Bash(npm run lint)",
      "Bash(npm test *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(git add *)",
      "Bash(git commit *)"
    ]
  }
}
```

> [!TIP]
> セッション中に権限確認のプロンプトで「Always allow」を選択すると、そのツールが `settings.json` の許可リストに自動追加されます。頻繁に使うツールは一度許可しておくと快適です。

> [!WARNING]
> `Bash(*)` のような広すぎる許可は避けましょう。想定外のコマンド（`rm -rf` や外部への通信など）まで自動実行されるリスクがあります。必要なコマンドを個別に列挙するか、`deny` で危険パターンを明示的に禁止するのが安全です。

## 典型的なワークフロー

Claude Code を日常の開発に組み込む際の、典型的なワークフローを紹介します。

### 新機能の開発フロー

1. **プロジェクトを理解してもらう**
   「このプロジェクトの構造を教えて」「認証周りのコードを説明して」など、まずプロジェクトの理解から始めます。
2. **計画を立てる**
   「ユーザープロフィール機能を追加したい。どういう実装方針がいい？」のように、まず計画を相談します。いきなり実装させるよりも、方針のすり合わせを先に行う方が手戻りが減ります。
3. **実装を依頼する**
   「その方針で実装して」と指示すると、Claude がファイルの作成・編集を実行します。変更ごとに diff を確認して承認します。
4. **テストと検証**
   「テストを実行して」「lint を通して」など、品質チェックを依頼します。エラーがあれば Claude が自動で修正を提案します。
5. **コミットと PR**
   「変更をコミットして」「PR を作成して」で Git 操作も Claude に任せられます。コミットメッセージも自動生成してくれます。

### 効率的に使うコツ

- **具体的に指示する**
  「コードを直して」より「`src/auth.ts` の `validateToken` 関数で JWT の有効期限チェックが漏れているので追加して」の方が正確な結果が得られます。
- **`CLAUDE.md` を活用する**
  プロジェクトのルールや規約を `CLAUDE.md` に書いておくと、毎回指示しなくても Claude が従ってくれます。
- **段階的に進める**
  大きなタスクは一度に全部頼まず、ステップに分けて依頼する方が品質の高い結果が得られます。
- **diff を確認する習慣**
  Claude の変更は必ず diff で確認してから承認しましょう。意図しない変更を早期に発見できます。

> [!CAUTION]
> Claude Code は強力ですが、最終的な責任は常に開発者にあります。テストが通っていても、生成されたコードをそのまま本番に流し込むのは避け、レビューとテストを経てからデプロイしてください。
