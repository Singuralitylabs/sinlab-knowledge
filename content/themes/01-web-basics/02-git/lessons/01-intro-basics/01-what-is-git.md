# Gitとは・バージョン管理の概念

## 解説

### バージョン管理とは

バージョン管理（Version Control）とは、ファイルの変更履歴を記録・管理する仕組みです。

ソフトウェア開発では、コードを書き換えるたびに「いつ」「誰が」「何を」「なぜ」変更したかを記録することが不可欠です。バージョン管理がなければ、以下のような問題が発生します。

- `report_最終版.docx`、`report_最終版_修正.docx`、`report_本当の最終版.docx` のようなファイル名による管理の破綻
- 変更を元に戻せない（どこを変えたか分からない）
- 複数人の作業が衝突し、誰かの変更が消える
- 「あのとき動いていたコード」に戻れない

バージョン管理システム（VCS: Version Control System）はこれらの問題をすべて解決します。

---

### 集中型と分散型

バージョン管理システムには大きく2つの方式があります。

#### 集中型（Centralized VCS）

代表例：**Subversion（SVN）**、**CVS**

```text
開発者A ──┐
開発者B ──┼── 中央サーバー（唯一のリポジトリ）
開発者C ──┘
```

- 中央サーバーに1つだけリポジトリがある
- 開発者はサーバーに接続してファイルを取得・更新する
- サーバーがダウンすると作業できない
- ネットワーク接続が必須

#### 分散型（Distributed VCS）

代表例：**Git**、**Mercurial**

```text
開発者A（ローカルリポジトリ）──┐
開発者B（ローカルリポジトリ）──┼── リモートリポジトリ
開発者C（ローカルリポジトリ）──┘
```

- 各開発者がリポジトリの完全なコピーを持つ
- オフラインでもコミットや履歴閲覧が可能
- サーバーがダウンしても作業を継続できる
- 高速な操作（ほとんどの操作がローカルで完結）

---

### Gitの3つの領域

Gitはファイルを3つの領域で管理します。これはGitを理解するうえで最も重要な概念です。

```text
┌──────────────┐    git add    ┌──────────────┐   git commit   ┌──────────────┐
│  作業ディレクトリ │ ──────────→ │  ステージング   │ ──────────→  │  リポジトリ    │
│ (Working Dir)  │             │ (Staging Area)│              │ (Repository) │
└──────────────┘             └──────────────┘              └──────────────┘
    ファイルを編集         コミットする変更を選択          変更履歴を永続的に記録
```

| 領域 | 役割 | 例え |
|------|------|------|
| 作業ディレクトリ | 実際にファイルを編集する場所 | 作業机 |
| ステージングエリア | 次のコミットに含める変更を選ぶ場所 | 荷造り台 |
| リポジトリ | 変更履歴が保存される場所 | 倉庫 |

---

### Gitの特徴と利点

| 特徴 | 説明 |
|------|------|
| **高速** | ほぼすべての操作がローカルで完結するため非常に速い |
| **ブランチが軽量** | ブランチの作成・切り替えが一瞬（ポインタの移動だけ） |
| **オフライン作業** | ネットワーク不要でコミット・ログ閲覧・ブランチ操作が可能 |
| **データの完全性** | SHA-1ハッシュによりデータの改ざんや破損を検出 |
| **分散型** | 各開発者がリポジトリの完全なコピーを持つ |
| **自由なワークフロー** | チームに合わせた柔軟な運用が可能 |

### Gitの歴史

Gitは **2005年** に **Linus Torvalds**（Linuxの生みの親）によって開発されました。

それまでLinuxカーネルの開発には「BitKeeper」という商用の分散バージョン管理ツールが使われていましたが、ライセンスの問題で使えなくなったことをきっかけに、Linus自身が新しいバージョン管理システムを作ることを決意しました。

Gitの設計目標は以下の通りです。

1. **高速であること** — Linuxカーネルのような巨大プロジェクトでも快適に動作
2. **分散型であること** — 中央サーバーに依存しない
3. **データの完全性** — 履歴の改ざんを防ぐ
4. **大規模プロジェクト対応** — 数千人の開発者が参加するプロジェクトを支える

現在ではGitは世界で最も広く使われるバージョン管理システムとなり、GitHub、GitLab、Bitbucketなどのプラットフォームとともにソフトウェア開発の基盤となっています。

git --version

# Gitのヘルプを表示

## よくある間違い

### 1. バージョン管理 = バックアップだと思う

```text
❌ 誤解：Gitはバックアップツールである
✅ 正解：Gitは変更履歴の管理ツール。バックアップとは目的が異なる

バックアップ → ファイルのコピーを別の場所に保存
バージョン管理 → いつ・誰が・何を・なぜ変更したかの履歴を記録
```

### 2. Gitを使えば自動的に保存されると思う

```text
❌ 誤解：ファイルを保存すれば自動でGitに記録される
✅ 正解：git add と git commit を明示的に実行する必要がある

ファイル保存（Ctrl+S） → 作業ディレクトリに保存されるだけ
git add → ステージングエリアに登録
git commit → リポジトリに記録
```

### 3. GitとGitHubを混同する

```text
❌ 誤解：Git = GitHub
✅ 正解：
  Git    → バージョン管理ツール（ソフトウェア）
  GitHub → Gitリポジトリのホスティングサービス（Webサービス）

GitHubはGitを使うサービスの1つ。GitLab、Bitbucketなども同様のサービス。
```

## 実用例

### なぜバージョン管理が必要なのか — 実際のシナリオ

#### シナリオ1：変更を元に戻したい

```text
月曜日：ログイン機能を実装 → コミット
火曜日：ログイン画面のデザインを変更 → コミット
水曜日：「やっぱり月曜日のデザインに戻したい」

→ Gitなら過去のコミットに簡単に戻せる！
```

#### シナリオ2：複数人での共同作業

```text
Aさん：ヘッダーのデザインを修正中
Bさん：フッターに新しいリンクを追加中

→ Gitのブランチ機能で互いの作業を邪魔せず並行開発できる！
```

#### シナリオ3：リリース後のバグ修正

```text
v1.0をリリース後、v2.0の新機能を開発中
ユーザーから「v1.0にバグがある」と報告

→ Gitならv1.0のコードに戻ってバグを修正し、
  v2.0の開発にも反映できる！
```
