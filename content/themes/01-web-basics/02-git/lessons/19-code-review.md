# コードレビューのベストプラクティス

## 解説

**コードレビュー**とは、他の開発者が書いたコードをチームメンバーがチェックし、フィードバックを行うプロセスです。GitHubではプルリクエスト（PR）を通じて行われます。

### コードレビューの目的

| 目的 | 説明 |
|------|------|
| バグの早期発見 | 本番に出る前に問題を見つける |
| コード品質の向上 | 読みやすく保守しやすいコードにする |
| 知識の共有 | チーム全体の技術力を底上げする |
| 設計の一貫性 | プロジェクト全体のコードスタイルを統一する |
| 学習の機会 | レビューを通じて互いに学び合う |

### レビューで見るべきポイント

```text
機能面：
  □ 要件を満たしているか
  □ エッジケースが考慮されているか
  □ エラーハンドリングが適切か

可読性：
  □ 変数名・関数名は分かりやすいか
  □ コメントは適切か（多すぎず少なすぎず）
  □ 複雑な処理に説明があるか

設計面：
  □ 適切に分割されているか（関数・クラス）
  □ 既存のパターンに従っているか
  □ 重複コードがないか（DRY原則）

テスト：
  □ テストが書かれているか
  □ テストケースは十分か
  □ テストが正しく動作するか

セキュリティ：
  □ 機密情報がハードコードされていないか
  □ SQLインジェクションなどの脆弱性がないか
  □ 入力値のバリデーションがあるか
```

## コマンドサンプル

### GitHub CLIでレビューを行う

```bash
# PRの差分を確認
gh pr diff 42

# PRにレビューコメントを付ける（承認）
gh pr review 42 --approve --body "LGTM! コードがきれいです。"

# 修正を要求するレビュー
gh pr review 42 --request-changes --body "いくつか修正が必要です。コメントを確認してください。"

# コメントのみのレビュー
gh pr review 42 --comment --body "いくつか質問があります。"

# PRのレビュー状態を確認
gh pr view 42
```

### ローカルでPRのコードを確認する

```bash
# PRのブランチをチェックアウト
gh pr checkout 42

# 差分を確認
git diff main...HEAD

# ファイル一覧を確認
git diff main...HEAD --stat

# 特定ファイルの差分
git diff main...HEAD -- src/login.ts
```

## フィードバックの書き方

### 良いフィードバックの原則

```text
1. 具体的に書く
   × 「これはダメです」
   ○ 「この変数名を 'userData' から 'currentUser' にすると、
      中身が何か分かりやすくなると思います」

2. 理由を説明する
   × 「ここを変えてください」
   ○ 「この関数は100行を超えているので、
      データ取得部分を別関数に切り出すと
      テストがしやすくなります」

3. 提案型で伝える
   × 「間違っている」
   ○ 「こういう方法もあるかもしれません：〜」
   ○ 「個人的にはこちらの方が読みやすいと思いますが、いかがでしょうか？」

4. プレフィックスを使う
   [must]     必ず修正が必要
   [should]   強く推奨する修正
   [nit]      些細な指摘（修正しなくてもOK）
   [question] 質問・理解の確認
   [praise]   良い点を褒める
```

### レビューコメントの例

```text
--- 良い例 ---

[must] ここでユーザー入力をそのままSQLに渡しています。
プリペアドステートメントを使ってSQLインジェクションを防ぎましょう。

[nit] 変数名 `d` は `data` にした方が分かりやすいかもしれません。

[praise] このエラーハンドリングのパターン、とても参考になります！

[question] この処理はログインのたびに実行されますか？
キャッシュを使うとパフォーマンスが改善するかもしれません。

--- 悪い例 ---

「なぜこんな書き方をするのですか？」
→ 攻撃的に聞こえる。背景を知りたいなら丁寧に質問する。

「全部書き直してください」
→ 具体的にどこをどう直すべきかを伝える。
```

## GitHubのレビュー機能

### インラインコメント

```text
PRの「Files changed」タブで：
1. コードの行にマウスを合わせる
2. 「+」ボタンをクリック
3. コメントを入力
4. 「Add single comment」または「Start a review」を選択
```

### サジェスション機能

コード修正の提案をそのまま適用できる便利な機能です。

````text
レビューコメントで以下のように記述：

```suggestion
const userName = getUserName(id);
```

レビューを受けた側は「Apply suggestion」ボタンを
クリックするだけで修正を適用できる。
````

### レビューステータス

```text
Comment    → コメントのみ（承認でも却下でもない）
Approve    → 承認（マージしてOK）
Request changes → 変更を要求（修正後に再レビュー）
```

## レビューを受ける側の心構え

```text
1. 感謝の気持ちを持つ
   レビューは時間を使って品質を上げてくれる行為。
   「ご指摘ありがとうございます」と伝える。

2. 指摘を個人攻撃と捉えない
   コードへのフィードバックであり、人格否定ではない。

3. 意図を説明する
   「なぜこの実装にしたか」を丁寧に説明する。
   良い議論のきっかけになることが多い。

4. 全ての指摘に対応する
   同意 → 修正して報告
   不同意 → 理由を説明して議論

5. 対応したら報告する
   「修正しました、再レビューお願いします」
```

## レビューチェックリスト

### PRを出す前のセルフチェック

```text
□ コードは動作確認済みか
□ テストは追加・更新したか
□ テストは全て通るか（ローカルで確認）
□ 不要なデバッグコード（console.log等）を削除したか
□ コミット履歴はきれいか
□ PRの説明は十分か
□ 差分を自分で見直したか
□ 関連するIssueをリンクしたか
```

### レビューする側のチェック

```text
□ PRの説明を読んで背景を理解したか
□ コードの目的が明確か
□ テストが十分か
□ エッジケースは考慮されているか
□ 命名は適切か
□ セキュリティ上の問題はないか
□ パフォーマンスに懸念はないか
□ 良い点も伝えたか
```

## よくある間違い

### 1. レビューなしでマージする

```text
悪い例：
  急いでいるから自分でマージ
  → バグの見落とし、品質低下

良い例：
  ブランチ保護ルールで「最低1人の承認」を必須にする
```

### 2. 些細な指摘ばかりする

```text
悪い例：
  スペースの位置やカンマの後の空白など細かい指摘ばかり
  → レビューが形骸化、チームの士気が下がる

良い例：
  コードフォーマッターやリンターで自動化する
  レビューは設計やロジックに集中する
```

### 3. 大きなPRを一度にレビューする

```text
悪い例：
  2000行の差分を一気にレビュー
  → 集中力が続かず、重要な問題を見落とす

良い例：
  PRは200〜400行程度に分割
  「小さいPRを頻繁にレビュー」が理想
```

### 4. 攻撃的なコメントを書く

```text
悪い例：
  「こんなコードを書くなんて信じられない」
  「初心者ですか？」

良い例：
  「ここはこういうパターンにすると、もっとシンプルになりますよ」
  「この部分について一緒に考えましょう」
```

## 実用例

### 自動化ツールとの併用

```text
人間のレビューと自動化ツールの使い分け：

自動化すべきもの：
  - コードフォーマット → Prettier、Black
  - リンティング → ESLint、Pylint
  - 型チェック → TypeScript、mypy
  - テスト実行 → CI/CD（GitHub Actions等）
  - セキュリティスキャン → Dependabot、Snyk

人間がレビューすべきもの：
  - 設計の妥当性
  - ビジネスロジックの正しさ
  - 命名の適切さ
  - アーキテクチャの整合性
  - ユーザー体験への影響
```

### レビュー文化の育て方

```text
1. 最初は小さなPRから始める
2. レビューの良い例を共有する
3. ペアプログラミングも取り入れる
4. 「レビューは学びの場」という文化を作る
5. レビューにかかる時間を工数として認める
```

## 実習

### 課題1：セルフレビューを実践しよう

自分が書いたコードに対して、上記のチェックリストを使ってセルフレビューを行ってみましょう。見つけた改善点をリストアップしてください。

### 課題2：レビューコメントを書いてみよう

以下のコードに対して、レビューコメントを3つ以上書いてみましょう。プレフィックス（[must]、[nit]、[praise]など）を使ってください。

```javascript
function f(x) {
  var d = fetch("http://api.example.com/users?name=" + x);
  var password = "admin123";
  console.log(d);
  if (d != null) {
    return d;
  }
}
```

### 課題3：レビューガイドラインを作成しよう

自分のチーム（または個人プロジェクト）向けに、コードレビューのガイドラインを作成してみましょう。以下の項目を含めてください。

- レビューの目的
- レビュー時に確認すべきポイント
- コメントの書き方のルール
- レビュー依頼から完了までの目標時間
