# 講師メモ：GitHubチーム開発ハンズオン

## 講座概要

| 項目 | 内容 |
|------|------|
| 対象 | Git基礎習得済みの初心者（ブランチ作成・切り替えまで学習済み） |
| 形式 | 個人ハンズオン（各自手を動かす） |
| 所要時間 | 45分（標準）/ 60分（チャレンジ込み） |
| 前提知識 | clone / push / pull / ブランチ作成・切り替え |

---

## 事前準備チェックリスト

### 講師が用意するもの

- [ ] 練習用リポジトリ（`github-team-practice`）をGitHub上に作成済み
  - `contributors/example.md` が存在すること
  - `.github/PULL_REQUEST_TEMPLATE.md` が設定されていること
  - 詳細は「練習用リポジトリ設計.md」を参照
- [ ] リポジトリURLを参加者に共有する準備
- [ ] 自分のアカウントで一連の操作を事前に通しで確認済み
- [ ] スクリーン共有の準備（ブラウザ + ターミナルを並べて表示）

### 参加者に事前に伝えること

- GitHubアカウントを持参すること
- ターミナル（macOS: Terminal、Windows: Git Bash）が使える状態にしておくこと
- 練習用リポジトリのURLを当日共有すること

## 各ステップの解説スクリプト

### 導入（5分）

**口頭で話すポイント：**

> 「前回はブランチの作り方と切り替えまで学びましたね。今日はその続きで、チームで開発するときの流れを体験します。」
>
> 「GitHubでのチーム開発では、**プルリクエスト（PR）** が中心になります。自分の変更をmainブランチに取り込んでもらうための"提案"のイメージです。」
>
> 「今日やることを図で説明します。（図を見せる）この流れをひとつひとつ体験してもらいます。」

**板書・スライドに出す内容：**

```text
GitHub Flow の流れ

① ブランチを作る（git switch -c）
② 変更してコミット（git add → git commit）
③ GitHubにpush（git push）
④ プルリクエストを作成（GitHub上）
⑤ レビューしてマージ（GitHub上）
⑥ ローカルに反映（git pull）
```

**問いかけ例：**

> 「みなさん、チームで同じファイルを編集したら何が起きると思いますか？」  
>（コンフリクトの話につながる導線として）

---

### Step 1：ローカルマージ体験（10分）

**目的**: マージという概念を手を動かして理解させる

**解説ポイント：**

> 「まず、GitHubを使わずにローカルだけでマージを体験します。PRでやっていることの本質はこれです。」

- `git merge` コマンドの出力の「Fast-forward」という表示に触れる
- 「分岐したブランチの変更がmainに取り込まれた」ことを `git log --oneline` で確認させる

**よくある詰まりポイント：**

| 症状 | 原因 | 対処 |
|------|------|------|
| `git switch` がエラー | Git が古い（2.23未満） | `git checkout` を使うよう案内 |
| ファイルが作れない | ディレクトリにいない | `pwd` で確認させる |

---

### Step 2：PR作成（15分）

**目的**: push → PR作成の操作を習得する

**解説ポイント：**

> 「自分のforkしたリポジトリに変更をpushして、そこからPRを作成します。」

- GitHub画面を共有しながら「Compare & pull request」ボタンの場所を示す
- PRテンプレートが自動的に入力欄に表示される点を説明する
- タイトルは「何をしたか」が一目でわかるように書くことを強調する

**実演のポイント：**
- Files changed タブでdiffが確認できることを見せる
- 「緑が追加した行、赤が削除した行」と説明する

**よくある詰まりポイント：**

| 症状 | 原因 | 対処 |
|------|------|------|
| `push` が rejected | リモートが新しい | `git pull` してから `git push` |
| 「Compare & pull request」が出ない | push直後に時間が経った | Pull requests タブ → New pull request |
| PRのbaseがfork元になっている | デフォルト設定 | base リポジトリを自分のforkに変更するか、そのままでもOKと伝える |

> **備考（base リポジトリについて）**  
> forkした場合、PRのbase は自動的にfork元（講師のリポジトリ）に設定されます。  
> 今回は自分のfork内で完結させるため、base を「自分のfork の main」に変更するよう案内してください。  
> または、fork ではなく直接 clone させて self-hosted で進める方法もあります（参加者数が少ない場合）。

---

### Step 3：レビュー・マージ（10〜15分）

**目的**: PRのレビュー操作とマージを体験する

**解説ポイント：**

> 「実際のチーム開発では、ここに他のメンバーがコメントを書いてくれます。今日は自分でコメントを付けてみましょう。」

- Files changed の行コメントの付け方を実演する
- レビューコメントのプレフィックス（`[must]` / `[suggest]` / `[praise]`）を紹介する
- マージ後に `git pull` が必要な理由を説明する

**板書ポイント：**

```text
GitHubでマージ ≠ ローカルに反映

GitHubでマージしても、ローカルは古いまま。
git pull でローカルを最新にすることを忘れずに！
```

**よくある詰まりポイント：**

| 症状 | 原因 | 対処 |
|------|------|------|
| Merge pull request が押せない | コンフリクトが発生している | PR画面の指示に従ってローカルで解消 |
| pull しても変化がない | mainにいない | `git switch main` してから `git pull` |

---

### チャレンジ：コンフリクト体験（10分・任意）

**目的**: コンフリクトの発生原因と解決手順を体験する

**解説ポイント：**

> 「コンフリクトは「失敗」ではなく、チーム開発では必ず起きることです。慌てずに解決できるようになりましょう。」

- コンフリクトマーカー（`<<<<<<<` / `=======` / `>>>>>>>`）の読み方を説明する
- VS Code を使っている参加者にはビジュアルな解決ボタンがあることを紹介する

---

### まとめ（5分）

**振り返りの問いかけ：**

> 「今日体験したGitHub Flowの流れを、誰かに説明できますか？」  
> 「PR を作るメリットは何だと思いましたか？」

**伝えておきたいメッセージ：**

> 「チーム開発で大事なのはコミュニケーションです。わかりやすいコミットメッセージ、丁寧なPRの説明、建設的なレビューコメント、これらが技術スキルと同じくらい大切です。」

## 補足資料

### GitHub Flow の要点（参加者に口頭で説明）

1. `main` は常にデプロイ可能な状態を保つ
2. 作業は必ずブランチを切って行う
3. PRを通してレビューを受ける
4. 承認後にマージ

### 今回使ったコマンドの一覧

```bash
git switch -c ブランチ名    # ブランチ作成と切り替え
git add ファイル名          # ステージング
git commit -m "メッセージ"  # コミット
git merge ブランチ名        # マージ（ローカル）
git push -u origin ブランチ名  # リモートにpush
git switch main            # mainに切り替え
git pull                   # リモートの変更を取得して反映
git branch -d ブランチ名    # ブランチの削除
git log --oneline          # コミット履歴の確認
```
